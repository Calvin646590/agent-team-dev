## 端到端工作流（kind 无关）

```
T0 启动
  入口三选一 → 触发 Coordinator skill 加载到主会话
  主会话按 {text, source, context} 协议执行 Coordinator skill
  → 检测 .claude/agents/ 是否存在（ADR-0023 必需）
  → 根据 source 决定是否派 RequirementGatherer subagent

T1 团队组建
  Coordinator 派生决策
  规则匹配 → 命中：直接派生
            → 模糊：LLM 降级判断
            → 已缓存：复用 decisions.md
  → 列出本次任务的 agent 集合

T2 任务拆解
  Scheduler 创建 .agent-team/tasks/<id>.md（含 depends_on, next_steps 等）
  + 维护 index.md

T3 执行（DAG + IsolationStrategy）
  Scheduler 解析 DAG
  无依赖 agent 并发启动 → IsolationStrategy.create_workspace()
  agent 在自己 workspace 干活
  完成 → IsolationStrategy.collect_artifacts() → Scheduler 同步到下游 workspace
  失败 → 重试 N 次（默认 3，指数退避）→ 仍失败 → Mediator 上报用户

T4 收尾（PublishStrategy）
  全部完成 → Scheduler 触发 Merger
  Merger 调用 PublishStrategy.assemble() → PublishCandidate
  PublishStrategy.present(candidate, mode)
    Commander → 用户审核每一步
    Observer → 仅 quality_gate 失败 / 重大决策时询问
  用户接受 → PublishStrategy.apply()
```

## 上下文预算原则（ADR-0036，C' 架构的命脉）

C' 架构把 Coordinator/Scheduler/Mediator 都放进**主会话**上下文，三者还会接力、所有 subagent 返回都回流主会话。主会话上下文是**稀缺资源**，真实规模（多子任务 + 重试 + 大产物）下会爆。因此立硬原则：

1. **持久状态一律落 `.agent-team/` 文件**（tasks / index / decisions / log / workspace），主会话**只持指针 + 摘要**，不吞全文
2. **subagent 返回也落盘**（写到自己的 task md / workspace），主会话读摘要决定下一步，不把大段产物吞进上下文
3. 摘要必含的关键字段：**产物路径、status、下一步**（保证不丢关键信息）
4. demo 时**实测主会话上下文增长曲线**，作为 C' 架构可扩展性的硬指标

> 这条原则覆盖所有框架组件草稿。与之冲突的旧表述（如"状态传递不必走文件中转"）以本原则为准。

## 策略抽象

### IsolationStrategy

| 实现 | 适用 kind | 工作区位置 | 同步机制 | 失败回滚 |
|------|----------|-----------|---------|---------|
| git-worktree | development | Claude Code 自动建 `.claude/worktrees/agent-<id>/` | 上游分支 merge 进 base → 下游 worktree 从更新后 base 派生（ADR-0042，实测） | 丢弃 worktree / 任务级回滚 |
| directory-fork | content / research | `.agent-team/forks/<owner>-<task-id>/` | rsync 上游产物到下游 fork（小文件实拷贝 / 大文件只读 symlink） | 丢弃 fork |
| none | office | 项目根目录直接操作 | 按 `none_concurrency` 决定（serial / file-parallel） | 写前快照到 `.agent-team/snapshots/`，失败时可回滚 |

**IsolationStrategy.none 的额外约束**（ADR-0027/0028/0029）：
- `none_concurrency: serial`（默认）—— 严格串行
- `none_concurrency: file-parallel` —— 无依赖任务可并发，前提是 strict 模式下 files_scope.write 不重叠校验通过
- 写产物前自动快照到 `.agent-team/snapshots/<task-id>-<ts>/`
- snapshots 默认保留 7 天

**同步与快照的大/小文件约定**（第十二轮，实现注意事项）：
- **小文件实拷贝保隔离**：`sync_from_upstream` 默认用 rsync 实拷贝（**不用 symlink**），避免下游改动污染上游
- **大文件只读引用省成本**：单文件超阈值（默认 50MB，可在 team-config 调）→ 改用**只读 symlink**（下游只读不写）+ 日志标注
- 同一约定适用于 IsolationStrategy.none 的快照：超阈值文件只记 hash + 路径，不真备份（ADR-0029）

### PublishStrategy（"怎么交付"）+ apply_policy（"要不要 / 怎么应用"）

ADR-0033 把这两件正交的事拆成两个字段：`publish_strategy`（产出形态）与 `apply_policy`（应用行为）。

**publish_strategy**：

| 实现 | 适用 kind | 产出形态 | apply() 语义 |
|------|----------|---------|-------------|
| pr-style | development | git diff + 变更说明 | 接受后 git merge 到目标分支 |
| overlay | content / research | 新文件覆盖旧文件 + history 备份 | 接受后覆盖文件，旧版进 history/ |
| direct | office | 产物文件 + 外部动作清单 | 真调外部（发邮件、写日历等） |

**apply_policy**（独立维度，任意 publish_strategy 可组合）：

| 取值 | 默认归属 kind | 行为 |
|------|--------------|------|
| auto-apply | （无默认，需显式开） | present 通过即自动 apply，不再问 |
| require-review | development / content / research | apply 前必须用户审核 |
| dry-run | office | apply() 仅"标记完成 + 通知用户"，**不真调** apply() 的副作用（ADR-0030）|

→ 用户可在 README team-config 显式覆盖默认 `isolation` / `publish_strategy` / `apply_policy`

**Merger 的冲突处理**（assemble 阶段，每种 IsolationStrategy 不同）：
- git-worktree → 各分支依次 merge 到 `agent-team/integration-<task>`，冲突按 `files_scope.write` 归属判定，归属不明则标 `conflict: manual` 交用户
- directory-fork → 按各子任务 `outputs` 显式声明归集（**ADR-0039**，不靠"末端 fork"推断）；不同 agent 出不同文件→无冲突；同一文件多 agent 声明→ `conflict: manual` 交用户
- none → serial 无冲突；file-parallel 已校验 write 不重叠；意外重叠用 snapshots 对照列出
- 详见 [agent-team/agents/merger.md](../agent-team/agents/merger.md)（plugin 实现，权威）。Merger 只 assemble + 解决冲突 + 产出 PublishCandidate，**present/apply 由主会话按 apply_policy 执行**

### Connectors（外部系统接入，office 场景必备）

ADR-0031：通过 MCP server 接入外部系统，plugin 自身不内置连接器。

```yaml
# README team-config 增加
connectors:
  - name: gmail
    mcp_server: "@anthropic/mcp-gmail"
  - name: excel
    mcp_server: "@anthropic/mcp-excel"
```

agent md 的 `tools` 字段可声明使用这些 MCP 工具，例如 `tools: [Read, Write, mcp__gmail__send_draft]`。

#### 推荐 MCP server 列表（office 场景常用）

plugin 不内置连接器，以下是 office/research 场景常用的外部能力方向，供用户按需挂载（具体 server 以社区/官方实际可用者为准）：

| 能力方向 | 用途 | 备注 |
|---------|------|------|
| 邮件（Gmail / Outlook / SMTP） | 发草稿、读收件箱 | dry-run 下只生成草稿，apply 才真发（ADR-0030 / 0033）|
| 表格（Excel / Google Sheets） | 读写 xlsx、汇总、对账 | office 财务/数据整理高频 |
| 日历（Google Calendar / Outlook Calendar） | 排期、写日程 | 不可逆动作，建议 dry-run 默认 |
| 文件存储（Google Drive / Dropbox / S3） | 拉取输入、归档产物 | 大文件注意 IsolationStrategy.none 的快照成本 |
| 文档（Notion / Confluence） | research 资料检索、报告发布 | overlay publish 可对接 |
| 数据库（Postgres / SQLite MCP） | development/research 直查数据 | 配 files_scope 之外的只读约束 |

> 选型原则：优先官方 / 高星社区 server；连接器可靠性依赖第三方实现质量（ADR-0031 风险项）。首版只在文档里给方向，不绑定具体包名。

```markdown
---
id: 02-email-service

owner: backend-developer
depends_on: [01-db-migration]
next_steps:                         # 动态字段，由 owner agent 完成时填写

  - id: 04-tests

- ...

## 进度日志
- 2026-05-16 14:30 创建 service skeleton

## 产物
- src/lib/email/sender.ts
```

| 模式 | 行为 | 切换 |
|------|------|------|
| Commander | 派生前确认、分工前批准、关键节点拦截 | `/team mode commander` |

## 失败处理（ADR-0020 / 0035 修订）

> 重试 loop 的执行者是 **Scheduler**（不是 Coordinator）。ADR-0025 把调度从 Coordinator 拆给 Scheduler，ADR-0035 正式定归属：Scheduler 跑 retry loop，用尽/超时才 escalate 给 Mediator，Mediator 只管"需要人介入"。Coordinator 只派生、不碰失败。

```
agent 失败 / 超时（默认 10 分钟无进度）
  ↓
Scheduler 自动重试（默认 3 次，指数退避 5s/10s/20s）
  ↓
仍失败 → Scheduler 交 Mediator，Mediator 上报用户
  ↓
用户选项：
  - 重派（清除 attempts 重新派 N 次）
  - 换 agent（强制换一个能力相近的角色）
  - 跳过（标记 skipped，下游决定能否继续）
  - 用户接手（切到 Commander，用户手动操作）
  - 任务级回滚（ADR-0041：回到任务开始前）
```

### 跨子任务事务性（ADR-0041）

- **apply 批量后置**：T3 执行阶段**不真正 apply 外部副作用**（不发邮件 / 不合主分支 / 不覆盖正式文件）；所有产物攒到工作区，统一由 T4 Merger assemble → 主会话按 apply_policy **一次性** apply。避免任务半途失败留下"半应用"状态。
- **任务级回滚**：本次任务所有子任务的写前 snapshot 汇总到 `.agent-team/snapshots/<task-id>/`；失败时 Mediator 提供"一键回滚到任务开始前"。
- office direct 默认 dry-run 已部分缓解（T3 只写 outputs/，apply 才动外部）。

## 接口规范

### Coordinator skill 触发协议（统一入口）

Coordinator 是 **skill**（ADR-0032），不是 subagent。所有入口都通过加载 Coordinator skill 到主会话来启动，skill prompt 内部按下列约定执行。**没有"接口"实现，只有"约定的输入/输出"**。

```yaml
# 输入约定（skill prompt 顶部应明确读取这些）
text: string                    # 用户原话
source: "natural" | "slash" | "auto"
context:
  project_dir: string
  mode: "commander" | "observer"
  kind: "development" | "content" | "research" | "office"
  team_config: <解析后的 README team-config>

# 输出约定（skill 最终给主会话留下的状态）
task_id: string                 # 本次任务 id（已写入 .agent-team/tasks/index.md）
status: "started" | "needs_clarification" | "rejected"
message: string                 # 给用户看的下一步说明
# 若 needs_clarification → Coordinator skill 已派 RequirementGatherer subagent，返回澄清问题
# 若 rejected → 写明原因（例如缺 .claude/agents/）

# 错误处理：skill 直接给用户可操作的提示
#   - NoAgentsDir → 提示运行 /team init
#   - InvalidConfig → 指出哪个字段错了
#   - InvalidAgentMd → 指出文件名与问题
```

> ⚠️ Source 与 Mode 在多处复用，下面的 `IsolationStrategy` 等仍按"概念契约"描述，**不代表存在对应类**。本项目自研编排是基于 skill prompt + 主会话工具调用实现，没有真正的 OOP 接口。

### PublishStrategy

> **执行者拆分**（ADR-0032）：`assemble` 由 **Merger subagent** 做（组装 + 解决冲突 + 产出 candidate）；`present` / `apply` 由 **主会话**（Scheduler 阶段）按 `apply_policy` + `mode` 执行（present 要与用户交互、apply 要落地，都不需要再派 subagent）。下面的 interface 仅为概念契约，不代表存在对应类。

```typescript
interface PublishStrategy {
  assemble(artifacts: ArtifactSet[]): Promise<PublishCandidate>   // ← Merger subagent
  present(candidate: PublishCandidate, mode: Mode): Promise<UserDecision>  // ← 主会话
  apply(candidate: PublishCandidate): Promise<void>               // ← 主会话
}
// 概念实现：PrStylePublish | OverlayPublish | DirectPublish
```
