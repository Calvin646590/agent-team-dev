# 2026-05-23 第十六轮：development 端到端 demo 实测（R-8 暴露 + 框架跑通）

> 阶段三第一个端到端 demo。用真 plugin（agent-team）+ 真 git 项目（demo/string-utils）跑 development 单场景，目标观察 R-1/R-2。结果：第一跑暴露重大风险 R-8，加强制入口后第二跑**完整跑通**。

## 任务
给 string-utils 加 `slugify(s)` + `slugifyUnique(strs)`（复用 slugify，重复 slug 加 -2/-3 去重）+ 测试 + README + 新建 CHANGELOG.md。
真实依赖：slugifyUnique→slugify；tester/doc-writer→实现完成。

## Run 1（自然语言入口）—— ❌ 框架被绕过，发现 R-8

输入"用 team 给 string-utils 加 slugify…"。主会话**没加载 coordinator skill**，直接"读文件→并行派 3 个业务 agent→改完主工作区"。
铁证：无 `.agent-team/`、无 worktree、改动直接落主工作区（无 Merger / 无待审 diff）。且**违反 DAG**——tester 与 developer 并行，tester 启动时无 slugify，撞竞态，靠侥幸蒙对。

→ **R-8：框架被主会话绕过**（plugin 装了+启用了+skill 都在、描述含"用 team"触发词，仍没触发）。根因：skill 自动触发是建议性的，主会话对简单任务直接自己干。比 R-2 更深（不是状态机跑歪，是根本不进状态机）。记入 09 风险册（🔴🔴）。

## R-8 应对
- 加 `/team start` slash command（plugin v0.1.1），命令体强制框架协议
- 换难任务（真实依赖 + 多文件 + 新建文件）
- demo 重置基线

## slash command 注册受挫
`/team` 与 `/agent-team:team` 均 "Unknown command"——重启后仍然。command 文件在安装目录、版本对、frontmatter 正常，但运行时不认。**疑似 plugin command 需 plugin.json 显式声明或 Desktop 命令发现机制有别**（待查）。
→ 退到 fallback：把等效强制指令**直接粘贴**进会话（不依赖命令注册），照样测 R-8 根本问题"显式强制能不能逼框架进去"。

## Run 2（粘贴强制指令）—— ✅ 框架完整跑通

主会话**完整 engage**，全程（已 trust-but-verify 核实磁盘）：
- 复杂度闸门：显式判定 3 角色 + 真实 DAG → 必须用 team
- 读配置：kind=development / mode=observer / scheduling=dag / isolation=git-worktree
- Scheduler 拆 4 子任务，写 `.agent-team/tasks/01-04.md` + index.md + log.md + decisions.md
- 建 `agent-team/integration` 分支
- task 01（developer，worktree）→ merge 进 integration
- task 02（developer，**depends_on 01**，worktree，含上游 slugify）→ merge 进 integration（ADR-0042 真实生效）
- task 03（tester）+ 04（doc-writer）**并发**（都依赖 02）（V-10 真实生效）
- Merger 出 PublishCandidate（pr-style diff，quality passed，0 冲突）
- **main 未动**，交付待审 diff，require-review 等用户 /team accept（R-7 apply 后置生效）

核实：main 仍 baseline `64ae6ef`；integration 3 commit（DAG 顺序清晰）；`.agent-team/` 全落盘；integration 测试 9/9 绿；main 工作区干净。用户手动 apply（merge integration→main）后 9/9 绿。

## R-1 / R-2 判读
- **R-1 上下文：中等任务可控**。主会话编排仅 ~20-30k（4 子任务）→ **状态落盘+只持摘要（ADR-0036）奏效，主会话没爆**。线性外推大任务（~20 子任务）主会话编排可能逼近 100k+，**大任务要盯**。Merger 最贵（21k，重读产物+重跑测试+全量 diff）= 明确优化点。
- **R-2 调度：这次可靠**。DAG 顺序正确、无漏/重/乱并行/status 错。**prompt 驱动+自检（ADR-0037）在 4 节点 DAG 成立**，验证"先 prompt 不急上代码"。大 DAG 待观察。

## 形成的决策
- **ADR-0044**：可靠入口 = explicit /team 强制指令；自然语言降级 best-effort（修订 ADR-0006/0022）

## Follow-up（待办）
1. **slash command 注册修复**——查 plugin command 是否需 plugin.json 显式声明 / Desktop 机制
2. **plugin 内 subagent（merger/req-gatherer）bare name 调不到**——Run 2 里 merger 回退用了 general-purpose+注入角色；查是否需 `agent-team:merger` namespace 或注册有别
3. **Merger 成本优化**——不全量重跑测试（信任 tester 结果，只做整体 gate 抽查）；不重读全部产物（用 git diff）
4. **大任务压测**——R-1/R-2 在 10+ 子任务 DAG 下的表现（决定调度内核是否上代码，ADR-0037）

## 结论
**架构在真实 development 任务里立住了**：R-8 可治（强制显式入口），R-1/R-2 中等规模可控，worktree 并发同步 + DAG + Merger + apply 后置全部真实生效。下一步：修 follow-up 1-2（让 /team 和 plugin subagent 正常）→ 大任务压测 →（过了）4 场景铺开。
