# 2026-05-16 第十轮：方案 C 验证结论 + 框架层形态调整

> 第九轮决定走方案 C（业务 agent 走项目级 `.claude/agents/`），用 07-验证套件 实测。
> 本轮记录实测过程、踩坑、最终结论。

## 验证过程（按时间顺序）

### 第一次尝试：plugin 装好了但 agents 没注册

用户安装 plugin 后报告"装好了"，让主会话用 Agent 工具调 `test-coordinator`，返回：
```
Agent type 'test-coordinator' not found. Available agents: claude, claude-code-guide, Explore, general-purpose, hello-business, Plan, statusline-setup
```

而项目级 `.claude/agents/hello-business` **是**在列表里的（基线 OK）。

排查文件系统发现：plugin 实际安装在 `~/.claude/plugins/marketplaces/local-desktop-app-uploads/agent-team-c-validation/`，`installed_plugins.json` + `settings.json.enabledPlugins` 都正确，但 plugin 的 agents 仍然没注册到主会话的 Available agents 列表。

### 坑 1：plugin.json 里的 `agents` 字段写法

初版 manifest：
```json
{
  "name": "agent-team-c-validation",
  "version": "0.0.1",
  "description": "...",
  "agents": ["agents"]
}
```

这个 `"agents": ["agents"]` 是第二轮调研时根据"manifest 可选字段列表"猜的写法。**实测会让 plugin 静默不注册任何 agent**（不会回落到目录自动扫描）。

修复：去掉 `agents` 字段，让 plugin 自动扫描 `agents/` 目录：
```json
{
  "name": "agent-team-c-validation",
  "version": "0.0.2",
  "description": "..."
}
```

重启 Desktop 后，`test-coordinator` 正常出现在 Available agents 列表。

**教训**：plugin manifest 的可选字段，未经实测的"该字段语义"不可信。Skill / Hook 字段也要警惕。

### 关键发现：subagent 不能拿到 Agent 工具

修好 manifest 后，让主会话调 `test-coordinator`，它真的被调到了，自报：

> 项目级 agents 目录存在 ✓
> hello-business.md 文件存在且格式正确 ✓
> **但**："I don't have access to an 'Agent' tool in my current tool set. The tools available to me are only Bash and Read."

而 `test-coordinator.md` frontmatter 明确写了 `tools: [Agent, Bash, Read]`。**运行时 Agent 工具被 strip 了**。

也就是说，Claude Code 的 subagent 模型存在一条全局规则：
> **subagent 不能再派生 subagent**

不论是 plugin 内的 subagent 还是项目级 subagent，frontmatter 写 `tools: [Agent, ...]` 也拿不到 Agent 工具。这是为了避免无限递归 / 上下文爆炸的保护。

### 方案 C 真正的死法

我们原本以为方案 C 的风险是"plugin agent 看不见项目级 subagent"（namespace 问题）。**实际死法不是这个**：

- ✅ plugin 内的 subagent **能看到**项目级 subagent 的文件（文件系统层面读得到）
- ❌ plugin 内的 subagent **拿不到** Agent 工具 → 任何 subagent 都不能调任何 subagent（不区分项目级/plugin/built-in）

如果 Coordinator 实现为 subagent，它就**根本派不出业务 agent**。方案 C 形式上立不住。

### 转向：方案 C'（主会话作为 Coordinator）

第二轮验证假设：**框架层逻辑跑在主会话上下文里**（= skill 形态），主会话保留 Agent 工具，依然可以动态派生项目级业务 subagent。

加了一个 skill `test-orchestrate` 到 plugin（v0.0.3），让主会话执行：
1. Bash 探查 `.claude/agents/`
2. Read `hello-business.md` 拿 frontmatter
3. 自报当前能用的工具
4. 用 Agent 工具调 `hello-business`
5. 给结论

实测结果：

| 步骤 | 结果 |
|------|------|
| skill 自动触发（用户说"测试编排"命中 description） | ✅ |
| Bash + Read 工作正常 | ✅ |
| 主会话自报"我现在的 tools 列表里有 Agent 工具" | ✅ |
| 用 Agent 工具调 hello-business，返回正确字符串 | ✅ |
| `hello from project-level subagent (PID=16220, cwd=...)` | ✅ |

**主会话作为 Coordinator 这条路通了。**

## 锁定的事实矩阵

| 调用形态 | 是否可行 |
|----------|---------|
| 主会话 → 项目级 subagent（Agent 工具） | ✅ |
| 主会话 → plugin 内 subagent（Agent 工具） | ✅ |
| 主会话 → plugin 内 skill（自动触发 / 显式 /） | ✅ |
| 项目级 subagent → 任何 subagent | ❌（拿不到 Agent 工具） |
| plugin 内 subagent → 任何 subagent | ❌（拿不到 Agent 工具） |
| plugin 内 skill 加载到主会话后能否用 Agent 工具 | ✅（本质是主会话的能力） |

## 对架构的影响

需要派别人干活的框架层逻辑 → 必须跑在主会话 → 必须实现为 **skill / slash command**，不能实现为 subagent。

具体到我们的 5 个框架层 agent：

| 组件 | 是否需要派人 | 新形态 |
|------|------------|-------|
| Coordinator | 是（派业务 agent） | **skill** |
| Scheduler | 是（按 DAG 循环派业务 agent） | **skill** |
| Mediator | 是（可能要换 agent 重派） | **skill / slash command** |
| RequirementGatherer | 否（只对话、返回任务描述） | subagent ✓ |
| Merger | 否（只整合产物、输出 PublishCandidate） | subagent ✓ |

业务 agent 依然走项目级 `.claude/agents/`。ADR-0011（项目级完全自定义）+ ADR-0023（agents/ 必需）**保留**，只是"业务 agent 目录"的正式位置改成 `.claude/agents/`（这本来在第九轮就提到了）。
