## 策略抽象总览

```
project README team-config:
  kind: development | content | research | office

每个 kind 的默认策略：
                     IsolationStrategy   PublishStrategy
  development        git-worktree        pr-style
  content            directory-fork      overlay
  research           directory-fork      overlay
  office             none                direct(+dry-run)

用户可显式覆盖：
  isolation: directory-fork   # 哪怕是 dev kind 也可换
  publish: dry-run            # 试跑模式
```

### IsolationStrategy 接口

```
trait IsolationStrategy {
  fn create_workspace(agent_name: str) -> WorkspacePath
  fn sync_from_upstream(downstream: WorkspacePath, upstream: WorkspacePath)
  fn collect_artifacts(workspace: WorkspacePath) -> ArtifactSet
  fn cleanup(workspace: WorkspacePath)
}

实现：
- GitWorktreeIsolation       基于 git worktree
- DirectoryForkIsolation      .agent-team/forks/<agent>/ 目录
- NoneIsolation               不隔离（适合无文件产物的任务）
```

### PublishStrategy 接口

```
trait PublishStrategy {
  fn assemble(artifacts: ArtifactSet) -> PublishCandidate
  fn present(candidate: PublishCandidate, mode: Commander|Observer)
  fn apply(candidate: PublishCandidate)  // 用户接受后调用
}

实现：
- PrStylePublish              git merge → diff → 用户审核 → push
- OverlayPublish              新版本覆盖旧版本，旧版备份到 .agent-team/history/
- DirectPublish               直接动外部资源（含 dry-run 模式）
```

## 跨场景走查样本：内容创作（写一篇行业报告）

> 验证"策略抽象"在非开发场景是否真的立得住

### 场景设定
- 项目目录：`industry-report-2026-q2/`
- 项目类型：`content`
- 任务："写一篇 2026 年 Q2 AI 行业回顾报告，包含趋势、关键事件、未来预测"
- agents/ 配置：
  ```
  agents/
  ├── researcher.md          # 资料搜集
  ├── analyst.md             # 数据分析
  ├── writer.md              # 文字撰写
  └── editor.md              # 校对润色
  ```

### T0 启动
- 用户在 industry-report-2026-q2/ 目录里：「帮我把 Q2 报告写出来」
- Plugin 检测 agents/ → Coordinator.handle("...", "natural", {project_dir, kind:"content"})
- Coordinator → RequirementGatherer 追问：
  - "Q2 重点关注哪几个细分？大模型？应用层？硬件？"
  - "目标读者是谁？投资人？工程师？管理层？"
  - "有篇幅要求吗？"
- 用户回答 → 需求明确

### T1 团队组建
- Coordinator 走两级派生
  - 规则匹配 README team-config.derivation_rules（"报告" → researcher + analyst + writer + editor）
- 派生 4 个 agent

### T2 任务拆解
- Scheduler 创建子任务：
  ```
  .agent-team/tasks/
  ├── 01-collect-sources.md      (researcher, depends_on: [])
  ├── 02-extract-events.md       (researcher, depends_on: [01])
  ├── 03-analyze-trends.md       (analyst, depends_on: [01, 02])
  ├── 04-draft-sections.md       (writer, depends_on: [03])
  ├── 05-final-edit.md           (editor, depends_on: [04])
  └── index.md
  ```

### T3 并行执行 (DAG 调度 + DirectoryForkIsolation)
- IsolationStrategy = directory-fork
- 为每个 agent 创建：`.agent-team/forks/researcher/`, `forks/analyst/`, ...
- Scheduler 按 DAG 触发：
  - researcher 先跑（无依赖）
  - researcher 完成 → DirectoryForkIsolation.sync_from_upstream() 把 researcher 的产物（资料 md）拷到 analyst/writer fork 里
  - analyst 跑 → 完成后同步到 writer
  - writer 跑 → 完成后同步到 editor
  - editor 跑

### T4 收尾 (PublishStrategy = overlay)
- Scheduler 触发 Merger（在 content kind 下，Merger 调用 OverlayPublish）
- OverlayPublish.assemble() → 把 editor fork 的最终产物组装成 `report-2026-q2.md`
- Mode 是 Observer 的话 → 直接覆盖项目根目录里的同名文件，旧版本备份到 `.agent-team/history/2026-05-16T1430-report-2026-q2.md`
- Mode 是 Commander 的话 → 先展示 diff（与上一版本对比，或如果是新文件就展示概要），用户接受后再覆盖

### 验证结论
✅ **策略抽象立得住**：
- IsolationStrategy.directory-fork 替代了 git-worktree，agent 之间产物隔离 + 同步机制照常运作
- PublishStrategy.overlay 替代了 pr-style，最终产物以"覆盖+备份"形式交付
- 整个 T0-T4 工作流模式不变，只是策略实现不同

⚠️ **暴露的细节问题**（留作下一轮）：
- "report-2026-q2.md" 这个最终文件名怎么确定？写在 README 里？还是 RequirementGatherer 问？
- editor 编辑的是 writer 产物的副本还是原文件？如果是副本，editor 改完怎么"覆盖"writer 产物？
- DirectoryForkIsolation 同步上游产物时，是浅拷贝还是符号链接？大文件怎么办？
