# 2026-05-16 第九轮：动态派生限制 + 三方案抉择

> claude-code-guide 调研结果回来，发现一个核心架构假设被击穿，
> Plugin 内 agent **不能动态派生未在 manifest 中预定义的 agent**

## 命中的验证项

V-3：**Coordinator 能用 Agent 工具动态派生**未在 plugin 里预定义的 agent** —— ❌ 失败

这个能力是 ADR-0011（项目级完全自定义业务 agent）+ ADR-0023（agents/ 必需）的技术前提。
现在前提不成立，必须调整。

## 影响范围

依赖"动态派生"的设计：
- ADR-0011 项目级完全自定义业务 agent（核心特征）
- ADR-0010 派生引擎（规则匹配 → 真正派生需要 Agent 工具能调到未预定义的 agent）
- ADR-0023 取消零配置，要求 agents/
- F-1 项目内自动激活
- F-3 派生引擎

**核心特征"配置驱动"的整个生命线被卡住。**

## 三个变通方案

### 方案 A：Plugin 自我注入 agents
**机制**：
- 用户运行 `/team init` 或类似命令时，让 plugin 把项目 `agents/` 下的 md **复制**到 plugin 自身的 agents 目录
- 之后 plugin 内的 Coordinator 可以正常派生这些 agent（因为它们现在是"预定义"的）
- 用户改了项目 agents 后必须 `/team reload`

**优点**：保留 plugin 单元的完整性
**缺点**：
- plugin 是否能在运行时改自己的 agents 目录？需要验证
- 跨项目用同一个 plugin 时容易污染（A 项目的 agents 留在 plugin 里被 B 项目误用）
- 同步成本：每次用户改 md 都得 reload

### 方案 B：通用 worker + 注入 prompt
**机制**：
- Plugin 预定义一个 `business-worker` agent（角色"空白"）
- Coordinator 用 Bash 读项目 `agents/<role>.md`，把 frontmatter + system prompt 拼成消息
- 调用 `business-worker` 时把这段"角色注入" 作为任务上下文传入

**优点**：plugin 单元干净，业务 agent 完全由项目控制
**缺点**：
- 所有业务 agent 的"人设"必须在 prompt 里现场注入，不利用 Claude Code 原生的 subagent 隔离
- 一个 business-worker 处理所有角色，name 字段失去意义，日志难看
- isolation: "worktree" 还能用，但 model 选择、tools 限制都要在 prompt 层模拟（不如原生干净）
- agent md 里的 `tools` 字段失效（business-worker 自己有什么工具，所有"业务 agent" 就只能用这些）

### 方案 C：业务 agent 走 Claude Code 项目级 .claude/agents/
**机制**：
- 项目里的 agents/ 实际上就是 `.claude/agents/`（Claude Code 原生项目级 subagent 位置）
- Claude Code 启动时自动加载项目级 subagent，把它们注册到当前 session
- Plugin 内的 Coordinator 通过 Agent 工具按 name 调用，**Claude Code 在 session 中查找 subagent 时会同时检查 plugin 预定义、用户级、项目级三处**
- 业务 agent 完全由项目控制，符合"项目级完全自定义"

**优点**：
- 最优雅，完全符合 Claude Code 既有约定
- 项目级 subagent 仅项目内可见，无跨项目污染
- agent md 完整生效：name、tools、model、frontmatter 全部原生支持
- 不需要 reload、不需要复制
- ADR-0009/0011/0023 全部保留，只是路径名从 `agents/` 改成 `.claude/agents/`

**缺点 / 待验证**：
- ❓ Plugin 内 agent (Coordinator) 调用 Agent 工具时，**能否触达项目级 .claude/agents/ 的 subagent**？这是关键技术问题，需要验证
- 即使能调用，"按 name 触达" 是否会和 plugin/user 级 subagent 冲突？（namespace 问题）
- 用户必须接受"agents 目录在 .claude/agents/"这个位置约定（轻微违背我们原本的"项目根 agents/"设计）

## 推荐：方案 C，B 兜底

**理由**：
- C 最符合 Claude Code 的生态约定（项目级 subagent 本来就是为这个场景设计的）
- 不需要 plugin 自我修改、不需要 prompt 注入、不污染跨项目
- 唯一风险是技术可行性（plugin 内 agent 能否调用项目级 subagent），但这是个**可以快速验证**的小问题

**如果 C 验证失败 → 退到 B**：
- B 是确定能跑的（plugin 内 agent 用 Bash + 预定义通用 worker）
- 代价是 tools 字段失效 + 日志混乱，但功能上能跑

**A 不推荐**：跨项目污染问题严重，同步成本高。
