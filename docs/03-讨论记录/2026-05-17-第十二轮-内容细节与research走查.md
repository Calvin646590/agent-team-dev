## 第一部分：内容场景遗留细节（第六轮留）

第六轮内容场景走查（写行业报告）暴露了 3 个未定细节，现逐一定：

### 细节 1：最终产物文件名怎么确定？

**问题**：`report-2026-q2.md` 这个最终文件名从哪来？

**结论**：三级来源，按优先级取第一个命中的：
1. **RequirementGatherer 澄清时问出来的**（如用户说"输出叫 q2-report.md"）→ 写进 `task_description` / `acceptance`
2. **项目 README team-config 的约定**（可选加一个 `output_naming` 提示，非必填）
3. **Merger 兜底生成**：按 `<kind>-<task-short>-<date>.md` 规则，并在 PublishCandidate 里标"文件名为自动生成，可改"

→ 不新增强制字段；优先靠 RequirementGatherer 在澄清阶段顺带问清楚（它本来就在收集 acceptance）。

### 细节 2：editor 改的是 writer 产物的副本还是原文件？

**问题**：directory-fork 下，editor 在自己的 fork 里编辑。它改的是 writer 产物的拷贝。改完怎么"覆盖"成最终态？

**结论**：
- editor 的 fork 在 DAG 同步阶段已经 rsync 拿到 writer 的产物（ADR-0018 sync_from_upstream）
- editor **就地编辑自己 fork 里的那份**（即 writer 产物的副本）
- editor 完成后，它的 fork 产物就是**流水线末端的最终态**（见 merger.md 的 directory-fork 冲突策略："主线产物取流水线末端 fork 版本"）
- Merger 收口时取 editor fork 的版本作为成品，OverlayPublish 再覆盖到项目根目录、旧版进 `history/`

→ 没有"改副本还是原文件"的纠结：每个 agent 只动自己 fork 内的副本，最终态由 DAG 末端 + Merger 决定。原文件（上一版）由 overlay 备份进 history，不丢。

### 细节 3：directory-fork 同步是浅拷贝还是符号链接？大文件怎么办？

**问题**：sync_from_upstream 把上游产物给下游 fork，用 cp / rsync / symlink？

**结论**：
- 默认 **rsync（实拷贝）**，不用 symlink
  - 理由：symlink 会让下游"看似独立实则共享"，下游一改就污染上游，破坏隔离语义
  - rsync 增量传输，重复同步成本可控
- **大文件阈值**（与 ADR-0029 快照的大文件处理对齐）：
  - 单文件超过阈值（建议默认 50MB，可在 team-config 调）→ 该文件改用**只读 symlink**（下游只读不写它）+ 日志标注
  - 既避免大文件重复拷贝，又因为"只读"不破坏隔离
- 这条同样适用于 ADR-0029 的 none 快照：超阈值文件只记 hash + 路径，不真备份

→ 形成一条通用约定：**小文件实拷贝保隔离，大文件只读引用省成本**。记入 02 架构 IsolationStrategy 实现注意事项即可，不新增 ADR（属实现细节）。

## 第二部分：research 场景单独走查

此前 research 只是"推断与 content 同构"（同样 directory-fork + overlay）。现在单独走一遍，确认是否真同构、有没有 research 特有的坑。

### 剧本设定

**项目**：`market-research-saas-onboarding/`（调研"SaaS 新用户引导最佳实践"）
- `kind: research`
- 输入：一个调研问题 + 若干参考来源（可能要联网查）
- 期望产出：
  - `findings.md`（结构化发现）
  - `sources.md`（引用清单）
  - `recommendations.md`（结论与建议）

**.claude/agents/ 配置**（init 的 research 骨架）：
```
.claude/agents/
├── researcher.md     # 检索资料、联网查、整理来源
├── analyst.md        # 提炼模式、交叉验证
└── writer.md         # 写 findings / recommendations
```

### T0-T2（与 content 一致）
- T0：RequirementGatherer 追问"调研边界？目标深度？要不要联网？引用格式？"
- T1：Coordinator 规则匹配"调研|research|分析" → researcher + analyst + writer
- T2：Scheduler 拆：
  ```
  01-gather-sources    (researcher, depends_on: [])
  02-cross-validate    (analyst,    depends_on: [01])
  03-extract-findings  (analyst,    depends_on: [02])
  04-write-report      (writer,     depends_on: [03])
  ```
✅ 与 content 同构。

### T3（directory-fork + DAG）—— research 特有点

| 检查项 | content | research | 结论 |
|--------|---------|----------|------|
| 隔离方式 | directory-fork | directory-fork | 同 |
| 链式同步 | researcher→…→editor | researcher→analyst→writer | 同构 |
| **联网检索** | 偶尔 | **高频**（researcher 要 WebSearch/WebFetch） | research 特有：researcher agent 的 `tools` 要含 Web 工具 / 检索类 MCP |
| **来源可信度** | 弱 | **强**（要记 source + 可信度，交叉验证） | research 特有：analyst 的 quality_gate 应含"关键结论有 ≥2 来源佐证" |
| 产物 | 单篇报告 | 多文件（findings/sources/recommendations） | 不同文件，Merger 按文件归集（不冲突）|

**发现的 research 特有点（都不击穿抽象，属配置层）**：
- R-1：researcher agent 需声明 Web/检索类工具（`tools: [Read, Write, WebSearch, WebFetch]` 或检索 MCP）—— 属业务 agent 配置，不是框架改动
- R-2：research 的 quality_gates 偏"来源/可信度"（如"每个关键结论 ≥2 来源"），content 偏"文笔/结构"—— 属业务 agent 配置
- R-3：`sources.md` 这种"引用清单"是 research 高频产物，Merger 的 directory-fork 策略已能处理（不同文件归集，无冲突）

### T4（overlay publish）—— 与 content 一致
- 多个产物文件分别 overlay 覆盖，旧版进 history/
- apply_policy 默认 require-review

### research 走查结论
✅ **research 与 content 在框架层完全同构**（directory-fork + overlay + DAG），抽象立得住。
差异全部落在**业务 agent 配置层**（Web 工具、可信度类 quality_gates、多产物文件），不需要框架改动，符合 ADR-0011"业务角色项目级完全自定义"。

→ 验收标准里的 research 场景从"推断同构"升级为"已走查确认同构"。
