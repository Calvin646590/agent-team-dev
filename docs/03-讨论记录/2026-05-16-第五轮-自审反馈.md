# 2026-05-16 第五轮：自审反馈（v0.1 设计）

> 用户要求"通读现有文档提反馈"，本轮以评审者视角对前四轮的设计做系统性挑刺
> 共 12 个问题，按严重性分三档

---

## 🔴 严重问题（影响架构根基，必须先解）

### S-1：愿景"通用" 与 整套设计"git/PR 偏向" 强冲突

**事实**：
- 愿景 (00) 强调"覆盖办公/开发/创作/分析全场景"
- 但 ADR-0014 worktree、ADR-0015 PR-style、F-3 worktree 创建、F-6 quality_gates、tester agent 等几乎全部围绕软件开发设计
- 验收标准 (01) 也明确"git 项目"

**含义**：
- 内容创作场景：没有 worktree、没有 PR、没有 lint/test
- 办公场景：连 git 都没有
- 当前设计 = "通用 plugin 名义 + 开发专用实现"

**待决方向**（二选一）：
- A. 诚实地把首版定义为"软件开发 agent team"，通用性留作 v2 演进目标
- B. 把"隔离策略"和"发布策略"抽象成可替换接口，开发场景用 worktree+PR，其他场景用"草稿目录+正式合并"等

→ 倾向 A（首版聚焦，先把 1 个场景做透），但需用户确认

---

### S-2：worktree 隔离 与 子任务依赖 互斥

**事实**：
- ADR-0014 选 worktree 全隔离
- 场景 B 工作流里：DB 迁移 → 后端 API → 前端调用，存在硬依赖
- 但 worktree 是某一时刻快照 —— 全部并发启动时，前端的 worktree 看不到后端 worktree 里刚写的 API

**ADR-0014 自己提到风险但没给解**：
> "每个 agent 看到的是某一时刻快照，与最终合并状态可能有偏差"

**实际正确模型**：DAG 调度 + worktree 隔离
- Coordinator 解析子任务依赖关系
- 无依赖的子任务可并发（独立 worktree）
- 有依赖的子任务等上游完成 → 同步上游产物到下游 worktree → 启动下游

**待补**：
- 子任务 md 里的 `depends_on` 字段（已有）必须真实驱动调度
- "同步上游产物"具体怎么做：rebase 上游分支？cherry-pick？merge？
- 02-架构里的工作流图需要重画

---

### S-3：Coordinator 职责过载（单点风险）

**当前 Coordinator 承担**：
1. 派生决策（规则匹配 + LLM 降级）
2. 任务调度（按依赖顺序触发 agent）
3. 维护 tasks/index.md
4. 处理 escalate_to 上报
5. 触发 Merger
6. 用户对话的入口（ADR-0010 蕴含）

**风险**：
- 单点：Coordinator 挂了整个 team 瘫痪
- 上下文爆炸：所有 agent 的状态、对话、决策都汇聚到 Coordinator
- 难以测试：耦合太多职责

**建议拆分**：
| 角色 | 职责 |
|------|------|
| Coordinator（精简） | 仅派生决策（规则+LLM 兜底） |
| Scheduler（新增） | 解析依赖、触发 agent、维护 tasks/index |
| Mediator（新增 / 可选） | 处理 escalate、用户介入交互 |

→ 框架层 agent 从 3 个 → 4-5 个，但每个职责更纯

## 🟡 中等问题（影响首版质量，可先做后改）

### M-4：`files_scope` 无原生强制机制

- Claude Code subagent 的 `tools` 字段限工具，**不限文件路径**
- 要强制只能用 PreToolUse hook 拦截 Read/Write/Edit
- 否则 `files_scope` 只是"约定"，不是"边界"

**待决**：
- 首版接受"仅约定"（agent 自觉、Coordinator 在 system prompt 里强调）
- 还是上 hooks（增加 plugin 复杂度）

→ 倾向"首版仅约定 + 文档明确说明"，hooks 留作 v2

---

### M-5：失败 / 卡死 / 超时机制完全缺失

- 02 架构里只在"待补充"列出
- 但首版上线必备：
  - 单 agent 跑超时（如 worktree 30 分钟无 commit）
  - 上报后 Coordinator 的处理选项（重派 / 换 agent / 升级用户）
  - 部分失败时的回滚策略

**待补**：专门一节"失败模式与恢复策略"

---

### M-6：`handoff_to` 是静态字段，可能不灵活

agent md 里写 `handoff_to: [tester]` 是写死的。但实际：
- backend-developer 写完功能 → 有时交 tester（要测）、有时交 frontend（要对接）、有时直接交 Coordinator（要决策）
- 下游不应该取决于 agent 类型，而取决于**当前任务的下一步**

**建议**：
- 把 `handoff_to` 改成"默认下游候选"（不是唯一）
- 由 Scheduler 在调度时根据子任务 md 的 `next_steps` 字段决定真实下游
- 静态字段保留但仅作为 hint

---

### M-7：三入口的"同一性"未规范

ADR-0006 说"三入口导向同一套底层逻辑"，但：
- 自然语言入口 → RequirementGatherer 还是 Coordinator？
- `/team start <任务>` 是否跳过 RequirementGatherer？
- "项目内自动激活" 是进入目录就激活？还是用户首次输入时才激活？

**待补**：定义统一入口接口
```
所有入口最终都调用 → Coordinator.handle(task_text, source, context)
其中 source ∈ {natural | slash | auto}
Coordinator 根据 source + task_text 决定是否走 RequirementGatherer
```

---

### M-8：N-4"零配置可跑" 与 ADR-0011"项目级完全自定义" 矛盾

N-4 说没有 `agents/` 也能跑（Coordinator 兜底）。
ADR-0011 说业务 agent 项目级完全自定义，plugin 不预置。

→ 那"零配置"模式下，Coordinator 派什么业务 agent？

**两条路径选一**：
- A. 取消 N-4，明确"必须有 agents/ 才能用"
- B. plugin 提供一组"通用业务 agent"作为零配置模式的兜底（如 generic-worker），破例承认 plugin 也带业务模板

→ 倾向 A（更符合"配置驱动"的纯粹性）

## 🟢 轻微问题（建议改进）

### L-9：用户模式切换 UX 模糊

- 切换时正在跑的 agent 是否中断？
- "Peer 模式 agent @user" —— Claude Code 没有原生 @user 概念，需要用什么机制实现（询问用户问题？）
- 三种模式的视觉/听觉反馈怎么做？

**建议**：首版只支持 Commander + Observer，Peer 模式 v2 再做（避免做出"看似支持实际跑不顺"的功能）

---

### L-10：Plugin 技术可行性未验证

- Plugin 内 agent 能否动态派生子 agent（而非只能用 plugin 预定义的）？
- Plugin 能否注册 `isolation: "worktree"` 的 Agent 调用？
- Plugin 能否在 Claude Code 里检测当前项目目录、读 agents/？

**建议**：在进入正式编码前，先做一个 hello-world plugin（Coordinator + 1 个写死的业务 agent + 1 个 slash command）跑通，验证基础能力

---

### L-11：N-3 性能指标过乐观

- "Plugin 启动 < 1s" —— 难达成
- "简单任务端到端 < 30s" —— 包括 RequirementGatherer 多轮、Coordinator 决策、worktree 创建、agent 干活、Merger 合并 → 30s 不现实

**建议**：删 N-3 数字指标，改成方向性目标
- "用户输入到 RequirementGatherer 首条回复 < 5s"
- "不刻意追求极致延迟，优先保证流程正确"

---

### L-12：Coordinator LLM 决策成本累积

每次模糊场景降级 → 一次 Coordinator LLM 调用 → 大型项目可能几十次

**建议**：
- Coordinator 维护"决策缓存"：相同任务模式 + 相同可用角色 → 复用上次决策
- 缓存写到 `.agent-team/decisions.md`
