# 2026-05-16 第三轮讨论：场景 B 走查 —— 软件开发任务

> 通过走查一个具体场景，反推配置 schema 和工作流。

## 剧本设定

**产品**：一个小型 SaaS 任务管理应用
- 技术栈：Next.js + TypeScript + PostgreSQL
- 当前规模：~50 个文件，已经有用户登录、任务 CRUD 功能
- 状态：本地开发，单人维护

**目标任务**：为应用增加"任务到期前邮件提醒"功能
- 需要决定提前多久提醒（业务决策）
- 需要后端定时任务
- 需要邮件模板
- 需要前端的"启用提醒"开关
- 需要数据库迁移
- 需要测试

**为什么选这个例子**：
- 跨前端/后端/数据库/外部服务（足够立体）
- 有明确产出（一个可合并的 PR）
- 边界清晰，不会发散
- Claude Code 单会话也能做，但用 agent team 应该能更并行、更分工、更稳

## 用户回答

| 问题 | 回答 |
|------|------|
| 项目预先需要哪些 md？ | **`agents/` 目录，每个 agent 一个 md** |
| 派生决策由谁做？ | **多级：先规则匹配，不明确交给协调者** |
| 角色定义层？ | **项目级完全自定义** |
| 任务输入形式？ | **渐进式对话** |

## 本轮关键洞察

四个回答内部一致，共同指向：**项目目录里的 `agents/` 就是这个 team 的"组织架构图"**。

但暴露了一个待解决的张力：

> **"项目级完全自定义" 与 "渐进式对话需要内置 agent" 矛盾**
>
> - 渐进式对话必然需要一个"需求收集 agent"，这是框架功能
> - 多级派生需要一个"协调者 agent"作为兜底，这也是框架功能
> - 这两类 agent 不可能让用户每个项目从头定义

**初步结论（待下一轮确认）**：
- **框架层 agent**（Coordinator、RequirementGatherer 等元层角色）由 plugin 提供
- **业务层 agent**（frontend/backend/dba/...）由项目 `agents/` 目录定义

## 推导出的 plugin 与项目目录结构（草案）

```
[Plugin 内置]
agent-team/
├── agents/
│   ├── coordinator.md          # 框架层：派生决策、调度
│   └── requirement-gatherer.md # 框架层：渐进对话、需求澄清
├── commands/
│   ├── team-start.md           # /team start <task>
│   ├── team-status.md
│   └── team-mode.md            # 切换指挥官/旁观者/同事
├── skills/
│   └── auto-activate.md        # 检测项目有 agents/ 时自动激活
└── (派生引擎、规则匹配器等代码)

[用户项目目录]
my-project/
├── agents/
│   ├── README.md               # 项目级元信息（项目背景、整体工作流）
│   ├── frontend.md             # 业务层 agent
│   ├── backend.md
│   ├── dba.md
│   └── tester.md
├── .agent-team/
│   ├── workspace/              # 共享文件区（agent 间产物）
│   ├── tasks.md                # 当前任务 / 子任务列表
│   └── log.md                  # 协作日志 / 审计记录
└── (项目源码)
```
