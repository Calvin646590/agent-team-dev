# 2026-05-22 第十三轮：V-11 / V-10 实测（阶段二最高风险项）

> 阶段二编码门槛验证的开场。先打两个最高风险项：V-11（skill 接力，决定框架层是 3 skill 还是 1 skill）、V-10（主会话并发调多 subagent + 收敛，T3 命脉）。
> 工具：扩展验证套件 plugin v0.0.4（test-relay-a/b + test-parallel）+ test-project 加 worker-1/2/3。

## V-11：skill 能否中途接力另一个 skill —— ✅ 通过

**测法**：触发 `test-relay-a` skill，让它用 Skill 工具调起 `test-relay-b`。

**实测结果**：
- 主会话自报"有 Skill 工具" ✅
- 从执行中的 test-relay-a 成功调起 test-relay-b ✅
- 看到 test-relay-b 的标记 `RELAY-B EXECUTED` ✅

**结论**：**一个 skill 能在执行中途用 Skill 工具调起另一个 skill。**
→ Coordinator → Scheduler → Mediator 三段接力成立，框架层**维持 3 个独立 skill**（ADR-0032 不变），无需合并成 1 个大 orchestration skill。S-1 悬念解除。

## V-10：主会话并发调多 subagent + 收敛 —— ✅ 通过（真并行）

**测法**：触发 `test-parallel` skill，在一条消息里并发派 worker-1/2/3（每个 sleep 2 秒），看是否都回、能否区分、是并行还是串行。

**踩坑（两次 inconclusive，根因同一个）**：
- 前两次 worker（和 hello-business）全报 `Agent type not found`，可用列表里没有任何项目级 subagent，只有 plugin 的 `test-coordinator` + 内置
- 一度怀疑是"worker 新建后没重启"，但**重启后 hello-business（早已验证可用）也消失**，排除该猜测
- **真根因**：项目级 `.claude/agents/` subagent 按**会话的项目根目录**注册，**不是 Bash 的 cwd**。当时的测试会话根不是 test-project（Bash 能 `ls` 到 worker 文件，但 Agent 注册表用会话根）
- **修复**：以 test-project 为项目根打开会话（与最早 C' 成功时一致）→ hello-business / worker 全部恢复可见

**修复后实测结果**：
| worker | PID | start_ts | end_ts | elapsed |
|--------|-----|----------|--------|---------|
| worker-1 | 5589 | 1779424719 | 1779424721 | 2s |
| worker-2 | 5593 | 1779424719 | 1779424721 | 2s |
| worker-3 | 5597 | 1779424720 | 1779424722 | 2s |

- 三个全返回，无丢失 ✅
- PID 各异（5589/5593/5597），无串台、无错配 ✅
- start_ts 三者相差 ≤1s，总跨度≈2-3s 而非 6s → **真并行**，不是串行 ✅

**结论**：**主会话能真正并发派多个项目级 subagent，并可靠收敛全部返回。**
→ 架构 T3 DAG"无依赖任务并行"成立，不是退化串行。

## 重要运行约束（V-10 副产，需记入设计）

> **项目级 `.claude/agents/` subagent 的注册基准是"会话项目根目录"，不是 Bash cwd。**

含义：
- 会话必须**以项目为根打开**，业务 subagent 才会注册、才能被主会话 Agent 工具调到
- Bash 的 `ls .claude/agents/` 能看到文件 ≠ Agent 工具能调到（注册表是会话启动时按项目根的快照）
- 改了 / 新增了 `.claude/agents/*.md` 后，可能需要重开会话才生效

对设计的影响：
- 印证 F-1"项目内激活"的前提（用户本来就在自己项目里工作，自然满足）
- `/team init` 生成 `.claude/agents/` 后，应提示用户"重开会话以加载新角色"
- 决策缓存的 `agents_hash`（ADR-0034）也得在会话级别重算
