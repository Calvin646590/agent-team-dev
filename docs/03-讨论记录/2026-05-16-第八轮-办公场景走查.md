# 2026-05-16 第八轮：办公场景走查

> 验证 IsolationStrategy=none + PublishStrategy=direct 是否真的立得住
> 这是设计 v0.3 剩下的最大未知

## 剧本设定

**项目**：`q2-finance-consolidation/`（季度财务表整理）
- 项目类型：`kind: office`
- 输入：`inputs/` 目录里有 5 个分公司 Q2 财务 Excel（结构略有差异）
- 期望产出：
  - `outputs/q2-consolidated.xlsx`（汇总表）
  - `outputs/q2-anomalies.md`（异常说明 —— 数据对不上的地方）
  - 一封发给财务总监的总结邮件草稿

**agents/ 配置**：
```
agents/
├── data-collector.md       # 扫描 inputs/，识别每个 Excel 的结构
├── data-cleaner.md         # 标准化、对账、找异常
├── consolidator.md         # 合并成汇总表
├── reporter.md             # 写异常说明 + 邮件草稿
└── README.md               # team-config + 派生规则
```

**README team-config**：
```yaml
kind: office
mode_default: commander       # 办公场景默认严格审核
publish: dry-run              # 默认不动外部
scheduling: dag
files_scope_enforcement: strict   # 涉及外部资源，硬约束
```

**为什么选这个例子**：
- 多 agent 都要操作 Excel 文件 → 暴露 IsolationStrategy=none 的并发问题
- 有"动外部系统"成分（邮件草稿）→ 验证 PublishStrategy=direct + dry-run
- 数据校验有"不确定性"（哪些是异常）→ 测试 quality_gates 在非代码场景下怎么定义

## T3 执行（IsolationStrategy=none + DAG）

**这里是最大压力测试。**

### 子问题 3.1：none 策略下 agent 的工作区在哪？

按 02 架构表："none → 项目根目录直接操作"

但场景 B 里 5 个 Excel 在 `inputs/`，4 个 agent 都要读 → 读没问题（读不冲突）。
写产物到 `outputs/` —— 谁先写？怎么不互相覆盖？

→ ⚠️ **发现问题 1**：none 策略下，"工作区"概念退化了。02 架构表里的同步机制写"无隔离 → 串行执行"，但这意味着完全放弃并发，DAG 的"无依赖任务并行"也失效。

**应对**：
- DAG 里 04 (anomalies) 和 03 (consolidate) 都依赖 02，可以并行。但 none 策略下要么强制串行（性能损失），要么需要"逻辑隔离"（按文件分区）

### 子问题 3.2：同时写 outputs/ 怎么办？

03 写 `outputs/q2-consolidated.xlsx`，04 写 `outputs/q2-anomalies.md`，05 写 `outputs/email-draft.md`
→ 不同文件，文件锁不冲突 ✅

但如果是"同一个汇总表的不同 sheet 都要 agent A 和 agent B 各自写"呢？
→ 真实冲突。需要"按文件分配写权限"。

→ ⚠️ **发现问题 2**：none 策略需要补充 "files_scope.write 不可重叠" 的强约束（strict 模式下）

### 子问题 3.3：失败回滚怎么办？

worktree 隔离失败时可以丢弃 worktree。
none 策略下 agent A 已经写了 outputs/foo.xlsx，agent B 失败，要不要回滚 A 的产物？

→ ⚠️ **发现问题 3**：none 策略需要"产物快照 + 回滚"机制
- 方案：每次写产物前先把现有版本备份到 `.agent-team/snapshots/<timestamp>/`
- 失败时 Mediator 提示用户："产物已部分生成，要回滚到 X 时刻吗？"

### 子问题 3.4：agent 调外部 API（如发邮件）怎么"隔离"？

reporter agent 要"准备发邮件"。none 策略下，是直接调邮件 API 吗？
→ 不行，PublishStrategy=direct + dry-run 默认应该挡住

→ ✅ **PublishStrategy.direct + dry-run 抽象救场**：执行阶段只生成"草稿文件"，发布阶段才真的调外部 API
- agent 在 T3 里只写 `outputs/email-draft.md`
- T4 由 PublishStrategy.direct 决定要不要真发

### T3 阶段小结

⚠️ **none 策略需要 3 个补充设计**：
- 补 1：DAG 在 none 下的退化策略（强制串行 vs 文件级并发）
- 补 2：files_scope.write 强约束（不可重叠）
- 补 3：产物快照 + 回滚机制

→ 不是抽象垮了，但 IsolationStrategy=none 的实现需要比 git-worktree / directory-fork 多做这三件事

## T4 收尾（PublishStrategy=direct）

PublishStrategy.direct 是新引入的策略，没在 content 场景验证过。

### 收尾流程
- Scheduler 触发 Merger
- Merger 调 DirectPublish.assemble() → 把 outputs/ 下所有产物打成 PublishCandidate
- DirectPublish.present(candidate, mode=commander) → 用户审核每个产物

### 子问题 4.1：dry-run 模式下，"应用"是什么意思？

- `outputs/q2-consolidated.xlsx` 已经写了 → 用户接受 = 不做事？还是 = 通知用户"已完成"？
- `outputs/email-draft.md` —— 用户接受 = 真的发邮件吗？还是 = 把草稿留给用户手动发？

→ ⚠️ **发现问题 4**：PublishStrategy.direct 在 dry-run 模式下的"apply 行为"必须明确
- 方案：dry-run 模式下，apply() 仅做"标记为已完成 + 通知用户"，不调外部
- 用户要"真发"必须显式切到非 dry-run 模式

### 子问题 4.2：邮件这种"动外部"怎么真的发？

dry-run → 生成草稿
非 dry-run → 实际发出
但"发邮件"需要凭据（SMTP 账号、API key）。这些 plugin 怎么管？

→ ⚠️ **发现问题 5**：动外部系统需要"凭据/连接器"层，目前架构没有

**应对**：
- 短期：通过 MCP server 提供"邮件/日历/Excel"等连接器，agent 通过 MCP 工具调用
- 长期：plugin 引入"connectors" 配置（项目 README 声明用哪些 MCP server）

→ 这是个**架构延伸**，不是 v0.3 的漏洞，是 office 场景必然要补的能力

### T4 阶段小结

⚠️ **PublishStrategy.direct 需要 2 个补充设计**：
- 补 4：dry-run 模式下 apply() 的明确语义
- 补 5：外部系统连接器（建议借 MCP）

## 走查结论

### ✅ 抽象立得住的地方
1. T0/T1/T2 流程与开发、内容场景一致 —— 框架共性扎实
2. PublishStrategy.direct + dry-run 概念正确，能挡住"误发邮件"等不可逆动作
3. agents/ 配置 + README team-config 能描述办公场景

### ⚠️ 发现的 5 个补充设计需求

| ID | 问题 | 严重性 | 处理建议 |
|----|------|--------|----------|
| O-1 | none 策略下 DAG 的退化方式（串行 vs 文件级并发） | 中 | 默认串行（简单），可选文件级并发（需 files_scope 不重叠） |
| O-2 | none 策略下 files_scope.write 强约束 | 中 | strict 模式下 plugin 校验 agent md 之间 write 范围不可重叠 |
| O-3 | none 策略下产物快照 + 回滚 | 中 | 每次写前备份到 .agent-team/snapshots/<ts>/ |
| O-4 | PublishStrategy.direct + dry-run 的 apply 语义 | 高 | dry-run 模式 apply() 仅"标记完成 + 通知"，不真调外部 |
| O-5 | 外部系统连接器（邮件/Excel/日历等） | 高 | 借 MCP server 实现，README 里声明 connectors |

### 🟢 抽象垮了的地方
**无**。office 场景没有击穿 IsolationStrategy + PublishStrategy 的核心抽象，但暴露了 none/direct 这两个具体实现需要补充的细节。

→ **结论：v0.3 的策略抽象在 4 场景下都立得住。office 场景的 5 个补充设计列入 ADR 即可。**
