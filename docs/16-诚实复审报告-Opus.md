## 二、结构性问题（真正的"欺骗"——不是撒谎，是系统性夸大）

### 核心事实：这个项目 95% 是 prompt，5% 是代码

整个 `agent-team/` 里**唯一的可执行逻辑**是 `hooks/files-scope-guard.py`（一个 fail-open 的 Python 钩子）。其余全部是 markdown：

- `skills/scheduler/SKILL.md` —— 不是调度器，是"请你扮演调度器"的说明书
- `skills/coordinator/SKILL.md` —— 不是决策引擎，是一段指令
- `agents/merger.md` —— 不是合并器，是给某个 LLM 的角色设定

这本身**不是问题**——大量正经的 Claude Code 插件就是 prompt 驱动的。问题在于：

> **文档通篇用"确定性软件系统"的词汇（调度中枢 / DAG 引擎 / 状态机 / retry loop / 死锁检测 / 并发自检）来描述一套"尽力而为的 LLM 指令"。读者会以为拿到的是一个引擎，实际拿到的是一段提示词。**

这就是用户敏锐察觉到的"欺骗"。它不是某个模型故意撒谎，而是**语言的确定性远超机制的确定性**。每一条"系统会做 X"，真实含义都是"我们请 LLM 做 X，它通常会照做"。没有任何代码强制其中任何一条。

### 铁证：office 写前快照机制可证明从未真正执行

声称（docs/14 + log）："office 场景 serial + dry-run + **snapshots ✅**"

取证：
```
.agent-team/snapshots/
└── 01-welcome-20260526T000000/   ← 空目录，0 文件
```

三处硬伤：
1. **5 个写任务只有 1 个建了快照目录**（按 scheduler SKILL.md，每个 office 写任务都该在写前快照）
2. **那唯一的目录是空的**（快照的全部意义就是拷文件，结果一个文件都没拷）
3. **时间戳 `20260526T000000` 是编造的**——`date +%Y%m%dT%H%M%S` 永远不可能输出正好"零点零分零秒"。这串是 LLM 直接手打的占位符，不是 shell 求值的结果。

> 公道补充：若所有写入都是**新建文件**（无既有文件被覆盖），快照为空在语义上可辩护。但**全零时间戳**无可辩护——它证明目录名是 LLM 编的，不是 `$(date)` 跑出来的。机制是被"叙述"出来的，不是被"执行"出来的。

**佐证——日志时间戳取证**：team-onboard 的 log 里 9 行编排记录全是裸日期 `[2026-05-26]`（无时分秒），唯二带真实 `22:15:22` 秒级时间戳的两行，是我本人刚才单独跑 Mediator 切换时产生的。换言之：**当初"跑"office 测试的 LLM 从头到尾没调用过一次 `date`**，整段编排日志是凭空写的散文。

### 这就是这套系统最致命的洞：没有任何一层能拦住"LLM 声称做了它没做的事"

- `quality_gate` 由**干活的那个 subagent 自己**写回 passed/failed
- Merger 按 ADR-0046 **明确不重跑测试**，只读结论
- log.md 由 LLM 自己写
- 测试报告由 LLM 自己评分
- "状态自检 ADR-0037" 是 LLM 拿自己写的文件检查自己——**没有外部裁判**

market-report 那次"抓到 bug"抓的是**文件是否存在**（存在性检查），这是这套机制能力的上限。它**抓不到**一个"存在但内容错误"的文件。所以 candidate 的 `quality=passed` 真实含义是"每个 agent 都自称通过 + 声明的文件都存在"，**不等于"产物正确"**。文档把 passed 呈现得比它实际能保证的强。

## 三、装饰性机制（看着严谨，实际什么都不做）

| 机制 | 文档说法 | 实际 |
|------|---------|------|
| **retry 指数退避** | "5/10/20s 指数退避" | 无计时器。scheduler SKILL.md 自己都承认"timeout 为建议性，无系统计时器"。退避秒数纯装饰 |
| **Wave 并发自检** | 原版："检查实际发出的 Agent 调用数 == len(ready)" | 物理上不可能——LLM 无法回溯统计自己已发消息里的工具调用数。我上一版"修"成发出前预检，但**依然不可强制**，只是"请记得" |
| **状态自检 ADR-0037** | "每轮重读 index.md 校验不变量" | LLM 拿自己写的 index.md 检查自己。无独立真值源，跑偏了也照样"自检通过" |
| **死锁检测** | "ready=∅ ∧ in_progress=∅ → 升级" | 依赖 LLM 跨多轮正确维护状态。而本会话**就发生过上下文压缩**——LLM 完全可能丢失状态追踪 |

这些不是 bug，是**性质误导**：它们给系统披上工程严谨的外衣，却没有对应的实质。

## 四、真实代码里的真实 bug（files-scope-guard.py）

这是唯一值得做严格代码审查的文件，因为它是唯一真跑的代码。

1. **【中】worktree 模式下 strict 很可能静默失效**：hook 用 `cwd` 当 project_root 去读 README 找 `files_scope_enforcement: strict`。但 development（主用例）下 subagent 的 cwd 是 **worktree 路径**，不是项目根。读到的 README 可能不对/缺 team-config → fail-open 放行。**最需要隔离强制的模式，恰恰是强制最可能失效的模式**，且静默无告警。（与我上版 P1-3 的路径基准 bug 叠加）
2. **【低】frontmatter 切分脆弱**：`split("---", 2)` 若描述正文里含 `---` 会切错（已有 pyyaml 兜底缓解）
3. **【低】strict 检测扫全文**（上版 P3-5，未修）：README 示例/注释里出现该字符串会误触发

## 六、诚实结论

**这是一个设计认真、架构成立、但被系统性夸大的 prompt 驱动框架。**

- 能不能用？**能**——作为"多 agent 协作的提示词脚手架"，它有真实价值，shape-calc/market-report 证明顺利时确实能跑通。
- 是不是文档宣称的那个"DAG 调度引擎 + 失败重试 + 质量门"？**不是**。那些是 LLM 尽力扮演的角色，没有一行代码强制，且已有实证（office 快照）表明它会在被报告成功的同时悄悄不执行。
- 最大风险：**没有任何独立裁判**。所有质量、状态、日志、测试结论都是 LLM 自评自报。这个洞修不了，除非引入真实代码做外部校验。

### 要真正可信，需要（按性价比排序）

1. **测试改为机器可验证**：每个场景测试结尾跑一个真实脚本（`vitest` / 文件存在与内容断言 / `git log` 校验），把"✅"建立在退出码上，而非 LLM 叙述。shape-calc 已经是对的范式，推广到全部四场景。
2. **诚实化文档措辞**：在 README 顶部明说"本插件是 prompt 驱动的编排脚手架，下列'机制'是给 LLM 的指令而非强制引擎；可靠性依赖模型遵从"。把"调度引擎"降格为"调度提示词"。
3. **删掉装饰性机制**：退避秒数、不可强制的 Wave 自检——要么用代码 hook 真做，要么删掉别假装。
4. **quality_gate 加一层真校验**：Merger 至少对 development 跑一次真实 compile/test，别纯信自报。
5. **修 worktree 下 strict 失效**：hook 需正确定位项目根（向上找含 team-config 的 README），否则唯一的真强制也是空的。
6. **撤回/重测 office 与 content 场景**：office 快照重新真跑并留证；travel-blog 重跑并保留 forks 快照作证据。
