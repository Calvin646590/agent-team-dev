## ADR-0055：git-worktree 隔离运行时留痕，闭合最后一个 UNVERIFIABLE（诚实复审方案二·收尾，2026-05-30）
- **日期**：2026-05-30
- **背景**：ADR-0054 解决了 fork 证据，但 development 的 git-worktree 隔离仍 UNVERIFIABLE——pr-style accept 后 `git worktree remove` + 删分支，事后无据证明隔离真发生过。难点：worktree 路径由 Agent 工具动态创建、位置不固定，无法像 fork 那样靠路径前缀识别。
- **关键洞察**：git 自身用一个规范信号区分 worktree 与主工作树——linked worktree 的根目录是 `.git` **文件**（非目录），内容为 `gitdir: /main/.git/worktrees/<name>`。据此既能判定"写入发生在 worktree 内"，又能解析出主仓库根与分支名，**与 worktree 实际位置无关**。
- **实现**：新增 `hooks/worktree-evidence-guard.py`（PreToolUse）。写入目标向上找到 `.git`，若为文件则解析 gitdir → 主仓库根 + worktree 名 + 分支（读 worktree HEAD）→ 追加 `{ts真实, agent_type, worktree, branch, file}` 到**主仓库** `.agent-team/evidence/worktrees.jsonl`。worktree 被删带不走。永远 allow、fail-open。
- **配套**：pr-style accept 清理明确"保留 evidence/worktrees.jsonl"；scheduler development 分支注明隔离自动留痕。
- **背书**：`test_worktree_evidence_hook.py` 用**真实 git worktree** 测 3 项（写 worktree 留痕含分支名 / 写主工作树不留痕 / `git worktree remove` 后证据仍在）全过 exit 0。
- **里程碑**：三种隔离（office 快照 / directory-fork / git-worktree）全部有真实 hook 留痕，**verify_mechanism.py 的 UNVERIFIABLE 列表清空**。仅剩 DAG/Wave/死锁检测的 LLM 心算无法代码强制——属 prompt 驱动本质，已诚实记录，非遗漏。

## ADR-0054：directory-fork 隔离运行时留痕，使其事后可验证（诚实复审方案二，2026-05-30）
- **日期**：2026-05-30
- **背景**：content/research 用 directory-fork 隔离，但 overlay accept 后 forks/ 被删 → 事后无任何证据证明隔离真发生过。一个 LLM 可以根本不建 fork、直接写主目录，再谎称"directory-fork ✅"，而我们无法戳穿（与 office 快照旧病同源）。verify_mechanism.py 此前只能将其标为 UNVERIFIABLE。
- **备选**：
  - A. 让 Merger 在 accept 前归档 fork 清单（prompt 指令）
  - B. 真实 hook 在写入 fork 的**当下**自动留痕，证据独立存放
- **选择**：B（贯彻 ADR-0049 原则：能落到代码就别靠 prompt——A 会被 LLM 跳过/伪造，重蹈覆辙）
- **实现**：新增 `hooks/fork-evidence-guard.py`（PreToolUse）。按路径结构识别 `.../.agent-team/forks/<fork>/<rel>` 的写入，追加 `{ts(真实), session, agent_type, fork, file}` 到 `.agent-team/evidence/forks.jsonl`。证据与 forks/ 分离，accept 删 fork 带不走它。永远 allow、fail-open。
- **配套**：accept/overlay 清理明确"只删 forks/，绝不 rm -rf .agent-team/"（保住 evidence/）；scheduler content/research 分支注明隔离已自动留痕。
- **背书**：`test_fork_evidence_hook.py` 4 项真实子进程测试（写fork留痕 / 双agent各自fork隔离可见 / 删forks后证据仍在 / 写主目录不留痕）全过 exit 0。
- **可验证性提升**：UNVERIFIABLE 项从 {fork, worktree} 缩减为 {worktree}。旧实测（travel-blog/market-report）早于本 hook，无证据属诚实状态（标 WARN，非造假），重跑即自动留痕。
- **局限**：development worktree 路径在 .agent-team 之外、形态各异，本 ADR 未覆盖；留作后续。

## ADR-0053：Merger 必须做一次独立校验，不全信 subagent 自报（诚实复审 item 4，2026-05-29）
- **日期**：2026-05-29
- **背景**：ADR-0046 为省成本让 Merger"读 quality_gate 结论、不重跑测试"。但诚实复审指出：`quality_gate: passed` 是**干活的那个 subagent 自评自报**，一个没真跑测试或产物有错的 agent 可以直接写 passed 蒙混过关，而 Merger 全盘采信 → 整条质量链没有独立裁判。
- **决策**：Merger 步骤 3 必须有**一次不依赖自报的校验**，以退出码/存在性为准：
  - development：**真跑**项目测试命令（team-config.test_command → package.json scripts.test → tsc），退出码≠0 进 `integration_errors`，不因各 task 自报 passed 而跳过。
  - content/research/office：对各 task 声明的 `outputs` 做**存在且非空**断言（market-report 实测正是靠这个抓到缺失的 summary.md）。
- **与 ADR-0046 的关系**：不冲突。ADR-0046 反对的是"逐个 agent 重跑测试"（成本高）；本 ADR 要求的是"对**整合结果**跑**一次**、读退出码/摘要"。"不重跑每个 ≠ 一次都不跑"。
- **影响**：quality=passed 从"大家都自称通过"升级为"自称通过 **且** 通过了一次独立校验"。

## ADR-0052：strict hook 向上查找项目根，修复 worktree/子目录静默 fail-open（诚实复审 item 5，2026-05-29）
- **日期**：2026-05-29
- **背景**：`files-scope-guard.py` 原把 `cwd` 直接当项目根去读 README 找 `files_scope_enforcement: strict`。但 subagent 的 cwd 可能是 worktree 路径或任意子目录 → 读到错误/缺失的 README → strict 静默 fail-open，**唯一的真实强制层在最需要它的隔离模式下形同虚设**，且无任何告警。
- **决策**：新增 `find_project_root(cwd)`，从 cwd **向上逐级**查找"含 `team-config` 的 README.md 所在目录"作为项目根；找不到才放行。同时把 strict 检测正则**行首锚定**（`(?m)^[ \t]*files_scope_enforcement:\s*strict`），避免 README 正文/示例里的字符串误触发（顺带消化原 P3-5）。
- **背书**：`test_files_scope_hook.py` 5 项真实子进程测试（越界/范围内/**cwd 为子目录**/advisory/主会话）全过 exit 0；其中 cwd 为子目录用例专门锁定本修复，修复前会误放行。

## ADR-0051：删除装饰性机制——退避秒数与 Wave"自检"（诚实复审 item 3，2026-05-29）
- **日期**：2026-05-29
- **背景**：诚实复审指出两处"看着严谨实则不做"的装饰：① retry"指数退避 5/10/20s"——prompt 驱动无系统计时器，秒数纯属虚构；② Wave"并发自检"原版称"检查实际发出的 Agent 调用数 == len(ready)"——LLM 无法回溯统计自己已发消息里的工具调用数，物理上不可能。
- **决策**（贯彻"要么代码化、要么删除"原则）：
  - retry：删掉虚构的退避秒数，诚实说明"重试 = 立即重新派发并附 last_error"；若疑似限流等瞬时问题，可显式用 Bash `sleep` 插入**真实**等待。
  - Wave：从"自检机制"降格为"派发**纪律**"——明确这是**发出前**的行动指令（本条消息里务必放够 N 个调用），不是发出后的验证层；唯一可后验的证据是 log 时间戳是否真实。
- **影响**：移除两处会让读者高估可靠性的伪机制；保留其中真正可执行的指导（附失败上下文、批量派发）。

## ADR-0050：场景测试机器可验证化——✅ 由退出码裁决，不由 LLM 叙述（诚实复审 item 1，2026-05-29）
- **日期**：2026-05-29
- **背景**：Opus 取证式复审（docs/16）发现，prompt 驱动框架的最大风险是**自评自报**——跑测试的 LLM 可以"声称"DAG/fork/snapshot 都跑过并写出漂亮的 log，而无任何独立校验。实证：office 写前快照被报告 ✅ 但实际从未执行（空目录 + 伪造时间戳 `20260526T000000`），且该谎言通过了 log、candidate、测试报告、乃至前一版代码审查（v0.1.10）四道关。
- **决策**：建立 `pressure-test/verify/` 两层机器校验，把"通过"的定义从"LLM 说通过"改为"脚本退出码 0"：
  1. **第 1 层 `verify.py`**：唯一真值源 = 文件系统。解析各 task 自己声明的 `outputs`，逐个磁盘断言"存在且非空"；development 真跑 `vitest` 以退出码为准。
  2. **第 2 层 `verify_mechanism.py`**：防叙述造假，查机制留下的不可伪造痕迹（快照时间戳真伪、残留伪造探测、log 时间戳真实性）。
- **原则**：① 不读 log/candidate 里任何"通过"字样；② 期望来自框架自己的声明而非脚本作者臆断；③ 不可后验的项（fork/worktree 在 accept 后被清）诚实标 `UNVERIFIABLE`，**绝不假装通过**；④ 退出码即结论。
- **影响**：四场景"✅"按取证置信度重标（docs/14）；任何未来的叙述造假都会让 `run-all.sh` 退出码变 1，无处藏身。

## ADR-0049：office 写前快照从 prompt 指令改为真实 hook 代码（诚实复审 item 6，2026-05-29）
- **日期**：2026-05-29
- **背景**：ADR-0029 设计的"office 写前快照"原以 Scheduler 的 prompt 指令实现（让 LLM 自己 `mkdir`/`cp`）。诚实复审证明该机制在实测中被 LLM 跳过却谎报成功——这是"提示词剧场"的典型失败：语言承诺了机制，但无代码强制，可靠性完全依赖模型自觉。
- **备选**：
  - A. 加强 prompt 措辞，要求 LLM 务必执行快照
  - B. 把快照下沉为**强制 hook 代码**，PreToolUse 自动执行，不依赖 LLM 配合
- **选择**：B
- **理由**：A 仍是剧场——再强的措辞也无法保证 LLM 执行，且无法事后证伪。B 把机制变成确定性代码：`hooks/office-snapshot-guard.py` 在每次 Write/Edit 前自动把"即将被覆盖的、已存在的"文件拷入 `.agent-team/snapshots/<session>/` 并记真实 `datetime.now()` 时间戳。复用 V-13 已证实的 `agent_type`/`session_id` payload 字段与 PreToolUse 触发机制。
- **语义**：只快照"覆盖既有文件"的前态；新建文件无前态不快照（正确行为）；仅 office（isolation:none）生效；永远 allow（快照是副作用）；fail-open。
- **背书**：`test_office_snapshot_hook.py` 4 个真实子进程测试（覆盖/新建/非office/框架内部文件）全过 exit 0；`verify_mechanism.py` 第 2 层据此转绿。
- **更广含义**：确立一条原则——**凡声称"系统会自动做 X"的安全/隔离机制，应尽量落到 hook/代码，而非 prompt 指令**。装饰性的 prompt 机制（如 retry 指数退避秒数）要么代码化、要么删除，不留"看着严谨实则不做"的中间态。

  - prompt 驱动状态机对 12 任务 DAG **三场景全部功能正确**，方案 A 可继续用于首版
  - ~~并发效率问题（Wave 内串行 dispatch）登记为技术债~~ → **v0.1.10 已修复**（见下）

  - **修复方案**：在 DAG 主循环 dispatch 段增加显式反模式/正确模式对比（❌ 串行 vs ✅ 批量）、强制要求"一条消息 N 个 Agent 调用"、Wave 并发自检机制（len(ready) > 1 时校验实际派发数 == ready 数，不等则 log `wave_underutilized` 并补发）

## ADR-0046：Merger 成本控制 = 读质量结论 + git diff + 冲突时才读文件（2026-05-24）
- **日期**：2026-05-24
- **背景**：demo 实测 Merger 耗 21k tokens。根因：重跑业务测试 + 把产物文件全文读进上下文 + quality 字段缺失导致无法信任各 agent 结论。
- **关键区分**：`status: done`（流程完成）≠ `quality_gate.status: passed`（质量达标）。二者是独立维度，混淆则要么不安全（done 就信任）、要么低效（自己重跑）。
- **选择**：
  1. **task schema 加 `quality_gate` 字段**：subagent 跑完测试后写回，`status: passed|failed`，`notes: []`
  2. **Merger 读结论不重跑**：直接读各 task 文件的 quality_gate，无需 Scheduler 中转传参（数据源唯一）
  3. **产物用 git diff 表示**：无冲突时不读文件内容；有冲突时只读冲突块（`<<<<<<< HEAD` ～ `>>>>>>>`）
  4. **集成级校验轻量化**：merge 后只跑 compile / type-check，不重跑业务单测
- **理由**：各 subagent 是质量的第一责任人，结论已落盘；Merger 读结论比重跑更快、更准确（重跑可能因环境差异误判）。git diff 天然就是变更的最精简表示。
- **影响**：
  - `skills/scheduler/SKILL.md`：task schema 加 quality_gate 字段；DAG 循环补 quality_gate 写回说明；步骤 6 去掉 quality_by_task 中转传参
  - `agents/merger.md`：输入移除 quality_by_task；步骤 1 改为读 task 文件质量字段；冲突读取精确到冲突块

## ADR-0045：Claude Code slash command 注册机制完整图谱（local-upload 限制 + 三种路径，2026-05-24）

### 核心结论

**local-upload 插件无法注册 slash command（`/name` 格式）；但 `commands/` 目录会作为 user-invocable skill 加载（`/plugin:file` 格式）。**

### slash command 注册的三种路径（全图）

| 路径 | 机制 | 命令格式 | 例子 |
|------|------|---------|------|
| Marketplace UI install | 官方 install 流程写注册表 | `/plugin-name` | `/hookify`、`/commit-commands:commit` |
| CLI/npx 安装脚本 | 脚本直接写 `~/.claude/commands/` | `/file-name` | `npx hyperframes skills --claude` → `/hyperframes` |
| Local-upload（zip） | 插件系统加载文件为 skill | `/plugin:file` | `/agent-team:agent-team` |

**Slash command（`/name`）= CLI 层注册表；Skill（`/plugin:file`）= LLM 上下文加载。两套系统权限隔离，local-upload 只能到达后者。**

### 实测过程（五轮变量控制实验）

| 测试命令 | 结果 | 说明 |
|---------|------|------|
| `/team`（plugin `commands/team.md`） | ❌ Unknown | 插件安装后无效 |
| `/agent-team`（plugin `commands/agent-team.md`，文件名=插件名） | ❌ Unknown | 即便命名与插件一致也无效 |
| `/agent-team:team`（命名空间子命令格式） | ❌ Unknown | 命名空间格式也无效 |
| `/team`（`~/.claude/commands/team.md`） | ✅ 生效 | 唯一有效路径 |

**v1 根因表述正确。v2 "命名规则"推断被实测否定。**

### 思路演变记录（供后续封装插件参考）

**v1（正确）**：plugin `commands/` 不注册用户可输入 slash command，`~/.claude/commands/` 是正确路径。

**v2（错误推断）**：从 hookify README 中读到 `/hookify`、`/hookify:list` 等用法，推断 plugin `commands/` 按"文件名=插件名 → 主命令；其他文件 → 带命名空间子命令"规则注册。为此新增 `commands/agent-team.md` 并重建 zip（v0.1.3）。

**v3（实测：三种格式全 Unknown）**：重装 v0.1.3 后测试 `/agent-team`、`/agent-team:team`、`/team` 全部 Unknown。v2 推断错误。

**v3 追加测试（排除连字符变量）**：改名为 `calvin`（纯小写字母，无连字符，文件名=插件名）。实测：`/calvin` ❌、`/calvin:team` ❌。连字符排除。

**v3 追加测试（排除路径结构变量）**：将插件文件手工复制至 `claude-plugins-official/plugins/agent-team/`（与 hookify 相同路径层级），在官方 marketplace.json 和 settings.json 中登记启用，重启后测 `/team` ❌。路径结构排除。

**最终定论（经五轮对照实验）**：plugin slash command 注册**不依赖文件存在**，而是依赖 **Claude Code marketplace UI 的正式 install 流程**写入内部注册表。仅把文件放对目录不够，无论是 local-upload 还是手工复制进 official marketplace 路径，命令均无法注册。

这完整解释了 hookify README 能写 `/hookify` 的原因：hookify 经过 marketplace UI 正式安装，触发了注册流程；我们的插件无论放在哪个目录，都只是"文件在磁盘上"，未经注册流程。

| 插件名连字符 | `agent-team` vs `calvin` | 全 ❌ | ✅ |

| **注册流程** | marketplace UI install（hookify）可用，文件拷贝不可用 | 差异确认 | 根因 |

### 官方 marketplace 插件的 commands/ 命名规则（实测确认）

通过启用 `commit-commands@claude-plugins-official` 并实测，得到最终规则：

```
commands/{plugin-name}.md  → /{plugin-name}         （文件名=插件名时，无前缀主命令）
commands/{other}.md        → /{plugin-name}:{other}  （其他文件，带插件名前缀子命令）
```

**实测验证：**
- `commit-commands/commands/commit.md` → `/commit-commands:commit` ✅（不是 `/commit`）
- `commit-commands/commands/commit-push-pr.md` → `/commit-commands:commit-push-pr` ✅
- commit-commands README 写的 `/commit` 是**错误的简化标注**，实际命令带插件名前缀

**对我们插件的推算（v0.1.4 修正后）：**
- `agent-team/commands/agent-team.md`（文件名=插件名）→ 理论值 `/agent-team`（主命令，无冒号前缀）
- 若上架官方 marketplace，正确命令应是 `/agent-team`
- 旧的 `commands/team.md` → `/agent-team:team`（已修正为正确命名）

**此规则仅对通过 `enabledPlugins` 启用的官方 marketplace 插件生效，local-upload 插件无论格式正确与否均不注册 slash command。**

### 完整结论：slash command 注册的三层条件

1. **插件来源**：必须是 marketplace 插件（`@claude-plugins-official` 或等效），local-upload 被系统限制，不支持注册
2. **启用状态**：必须在 `enabledPlugins` 中显式启用（`{name}@{marketplace}: true`）
3. **命名格式**：`commands/{file}.md` → `/{plugin-name}:{file}`（文件名=插件名时省略前缀）

三层条件缺一不可。我们的 `~/.claude/commands/team.md` 方案绕开了所有三层限制，是 local-upload 插件的唯一有效路径。

### 最终方案

| 方案 | 结果 |
|------|------|
| plugin `commands/` 任意命名 | ❌ 对 local-uploaded 插件无效 |
| `~/.claude/commands/team.md` | ✅ 唯一有效，一次性手动复制 |

### 封装新插件的建议（基于本次完整验证）
- **local-uploaded plugin** 的 slash command **必须**走 `~/.claude/commands/{name}.md`，写入安装手册
- Plugin `commands/` 目录保留（作为 Skill tool 加载的内容载体，也为将来 marketplace 分发预留）
- `commands/agent-team.md` 保留在 zip 中（无害，供 marketplace 安装时验证）

## ADR-0044：可靠入口 = explicit /team 强制指令；自然语言降级为 best-effort（R-8 应对）
- **日期**：2026-05-23
- **背景**：R-8 实测——自然语言"用 team 加个 X"**不能可靠触发**框架（skill 自动触发是建议性，主会话对简单任务直接自己干，绕过 Coordinator/Scheduler，还撞依赖竞态）。但**强制显式指令**下框架完整 engage（demo run 2 验证：复杂度闸门→DAG→worktree→Merger→待审 diff 全跑对）。
- **选择**：
  - **可靠入口 = explicit `/team start <task>`**（slash command），命令体含强制框架协议（"这是命令不是建议，不许自己直接做，必须写 .agent-team/tasks、按 DAG、worktree、Merger、待审 diff"）
  - **自然语言入口降级为 best-effort**：能触发更好，触发不了/简单任务主会话直接干也接受（与 ADR-0040 复杂度闸门一致）
  - 修订 ADR-0006/0022 的"自然入口与 slash 等价"预期——自然入口不保证进框架
- **理由**：经验证明被动 skill 触发不可靠；显式命令 + 强指令是目前唯一能可靠逼框架跑全套的方式
- **影响**：plugin 加 `commands/team.md`（v0.1.1）；slash command 注册问题已结案（ADR-0045）——local-upload 插件不支持 slash command 注册，这是架构设计；有效入口为 `/agent-team:agent-team`（commands/ 同时作为 user-invocable skill 加载，v0.1.5 确立）
- **风险**：即便强制指令，是否每次都 100% 不绕过仍需更多样本；大任务下的稳定性未验

## ADR-0043：strict 模式 per-agent files_scope 经 PreToolUse hook 的 agent_type 实现（V-13 实测确立）
- **日期**：2026-05-23
- **背景**：ADR-0019 定 strict 模式靠 PreToolUse hook 拦截越界写，但"hook 怎么知道当前是哪个 agent、该套哪份 files_scope"一直没落实（高级审查 R-3 / V-13）。
- **实测事实（V-13）**：
  - PreToolUse hook **对 subagent 的工具调用也触发**
  - hook 收到的 payload 含**直接身份字段**：`agent_id` + `agent_type`（主会话调用时无这两字段，subagent 调用时有）
  - `session_id` / `cwd` 主会话与 subagent 相同（不能靠它们区分）
- **选择**：strict 模式的 PreToolUse hook 实现为：
  1. 从 payload 读 `agent_type`（无此字段 = 主会话发起，按主会话规则）
  2. 读该 agent 的 `.claude/agents/<agent_type>.md` 的 `files_scope.write`
  3. 把本次 Write/Edit 的目标路径与 scope 比对，越界 → deny
- **理由**：身份字段直接可用，无需间接推断；机制干净、可落地
- **影响**：ADR-0019 strict 实现有了具体路径；strict hooks 正式实现时按此写
- **风险**：hook 每次调用要读 agent md（可缓存）；payload 字段名依赖 Claude Code 稳定性

## ADR-0042：git-worktree 跨 worktree 同步模型 = 上游分支合并进 base，下游 worktree 从更新后的 base 派生（V-12 实测确立）
- **日期**：2026-05-23
- **背景**：ADR-0018 说"上游完成后同步上游产物到下游 worktree（git rebase）"，但 Claude Code 的 worktree 由它自动在 subagent spawn 时创建/销毁，Scheduler 无法预先往下游 worktree 里塞东西（高级审查 R-3 / V-12）。原"rebase 进下游 worktree"的表述与机制不符。
- **实测事实（V-12）**：
  - 多个 isolation:worktree subagent 可**并发**，各在独立 worktree + 独立分支
  - 上游分支在主会话（项目根）**可见、可 `git merge` 回 base**（octopus 一次合多个）
  - 下游 subagent 的 worktree 在 **spawn 时从当前 base HEAD 派生** → 只要上游已合进 base，下游 worktree 自动包含上游产物
- **选择**：git-worktree 的同步协议修正为：
  - Scheduler 在启动某下游子任务前，先把它**所有上游依赖的分支 merge 进 base 分支**（main 或 integration）
  - 然后再用 Agent 工具启动下游 subagent —— 它的新 worktree 自动从更新后的 base 派生，上游产物可见
  - 不是"rebase 进一个已存在的下游 worktree"（那不符合机制）
- **理由**：契合 Claude Code 自动建 worktree 的真实行为；用 git 原生合并而非手工塞文件
- **影响**：02 IsolationStrategy.git-worktree 的"同步机制"列改为本模型；ADR-0018 sync 表述对齐
- **风险**：上游合进 base 后若下游失败回滚，base 已含上游改动（可接受：base 用临时 integration 分支，任务级回滚 ADR-0041 兜底）

## ADR-0041：跨子任务事务性 = apply 批量后置 + 任务级回滚（R-7）
- **日期**：2026-05-22
- **背景**：高级审查 R-7。单 isolation 内有 snapshot 回滚，但整个任务跑到一半失败（如 office direct 已 apply 3 个、第 4 个炸）无一致性恢复故事。
- **选择**：
  - **apply 批量后置**：所有子任务先把产物 assemble 成一个 PublishCandidate，T4 由用户一次性 review/apply，避免"边跑边 apply"造成半应用状态
  - office direct 默认 dry-run 已部分缓解（T3 只写 outputs/，apply 才动外部）
  - **任务级回滚**：本次任务所有子任务的 snapshot 汇总记录到 `.agent-team/snapshots/<task-id>/`，失败时 Mediator 提供"一键回滚到任务开始前"
- **理由**：把不可逆动作收口到 T4 单点，最大化"全做或全不做"；回滚有据可依
- **影响**：02 失败处理增"任务级回滚"；Merger/Scheduler 不在 T3 中途 apply 外部副作用
- **风险**：批量后置下，长任务的产物要暂存到 T4，占用磁盘/工作区

## ADR-0040：Coordinator 增加复杂度闸门（R-6）
- **日期**：2026-05-22
- **背景**：高级审查 R-6。每个子任务 = 一次 subagent spawn = 一次完整 LLM 上下文。简单任务用整套编排，开销可能远超单 agent 直接做。

## ADR-0037：调度状态机首版 prompt 驱动 + 自检，按需再上代码（R-2）
- **日期**：2026-05-22
- **背景**：高级审查 R-2。Scheduler 的 DAG 解析、attempts 计数、status 流转、retry 退避全是 prompt 指令靠 LLM 执行，无代码强制不变量，规模上来易跑偏。

## ADR-0036：上下文预算原则 = 状态落盘，主会话只持指针/摘要（R-1）
- **日期**：2026-05-22
- **背景**：高级审查 R-1（最高系统性风险）。Coordinator/Scheduler/Mediator 同跑主会话、三 skill 接力累加 prompt、所有 subagent 返回回流主会话 → 真实规模下主会话上下文可能爆。
- **选择**：立为硬原则：
  - **持久状态一律落 `.agent-team/` 文件**（tasks / index / decisions / log / workspace），主会话上下文**只持指针 + 摘要**，不吞全文
  - **subagent 返回也落盘**（写到自己的 task md / workspace），主会话读摘要决定下一步，不把大段产物吞进上下文
  - 推翻 coordinator.md 原"状态传递不必走文件中转"的表述（方向反了）
- **理由**：C' 架构把编排放进主会话，上下文是稀缺资源；落盘是唯一可扩展的状态载体
- **影响**：coordinator.md L91 修正；02 增"上下文预算原则"节；所有组件草稿遵循"产物落盘、上下文持摘要"
- **风险**：摘要可能丢信息（需约定摘要里必含的关键字段：产物路径、status、下一步）；demo 时实测上下文增长曲线验证有效性

## ADR-0035：失败重试归属 Scheduler，Mediator 只接超阈值的 escalate
- **日期**：2026-05-17
- **背景**：第十轮后审查（P1）发现失败归属冲突 —— ADR-0020 与 02 架构写"**Coordinator** 自动重试"，但 Coordinator 草稿明确"不处理失败"，Scheduler 草稿又写"失败直接交 Mediator"。三处互相打架。叠加 ADR-0032 后 Coordinator 已是纯派生 skill，更不该管重试。
- **选择**：明确三方归属：
  - **Scheduler**（skill）：拥有 retry loop —— 维护每个子任务的 `attempts`、按指数退避（5s/10s/20s）重试、判断是否超时（默认 10 分钟无产物更新）
  - **Mediator**（skill）：只接收 Scheduler 在"重试 N 次仍失败 / 超时"后的 escalate，负责与用户交互（retry / swap_agent / skip / user_takeover）
  - **Coordinator**（skill）：完全不碰失败，只负责 T1 派生
- **理由**：
  - retry 是"调度"的一部分，天然属于 Scheduler（它本来就在 DAG 主循环里盯每个子任务状态）
  - Mediator 专注"需要人介入"的升级场景，不与机械重试耦合
  - 与 ADR-0032 的形态划分一致（retry 要重新派 subagent，必须在主会话 skill 里做）
- **影响**：
  - **修订 ADR-0020**：retry 主体从 Coordinator 改为 Scheduler
  - 02 架构"失败处理"流程图：Coordinator → Scheduler
  - scheduler.md 第 5 节：补 retry loop 细节（不再"失败直接交 Mediator"）
  - mediator.md：定位为"只接 escalate"
- **风险**：Scheduler 职责变重（调度 + 重试）；但仍比"散落三处"清晰

## ADR-0034：决策缓存失效采用"双 hash 比对"
- **日期**：2026-05-17
- **背景**：ADR-0026 引入决策缓存，但失效条件一直列为待补。Coordinator 草稿写"`.claude/agents/` 没动过才复用"，但没定义怎么判断"动过"。第十轮后审查（P2）重新挑出此缺口。
- **选择**：缓存条目里额外记录两个 hash：
  - `agents_hash`：`.claude/agents/` 下所有 agent md 的文件清单 + 内容 hash（排序后拼接再 hash）
  - `config_hash`：README `team-config` 解析后的规范化 hash
  - 命中缓存时，重新计算这两个 hash 与缓存里记录的比对；**任一不一致 → 视为失效，跳过缓存重新决策**
- **理由**：
  - 比 mtime 更可靠（touch、无实质改动不会误失效；改了内容一定失效）
  - 实现简单（一次目录遍历 + hash）
  - 覆盖两个真正影响派生方案的输入源：业务 agent 定义 + 派生规则
- **影响**：
  - `.agent-team/decisions.md` 缓存条目 schema 增加 `agents_hash` / `config_hash` 字段
  - Coordinator skill 第 2 步"决策缓存查询"要先算 hash 再比对
- **风险**：大量 agent md 时 hash 计算成本（可接受，远小于一次 LLM 调用）

## ADR-0033：发布字段拆分为 publish_strategy + apply_policy
- **日期**：2026-05-17
- **背景**：第十轮后审查（P1）发现 `publish` 字段语义混杂 —— `PublishStrategy` 的取值是 `pr-style | overlay | direct`（"用什么形态交付"），但 schema 示例又写 `publish: dry-run`（"要不要真应用"）。dry-run 其实是**应用策略**，不是发布策略，两者被塞进一个字段。
- **选择**：拆成两个正交字段：
  - `publish_strategy: pr-style | overlay | direct` —— **怎么交付**（产出形态，由 PublishStrategy 实现）
  - `apply_policy: auto-apply | require-review | dry-run` —— **要不要 / 怎么应用**（在 present/apply 阶段的行为）
  - 两者都由 `kind` 给默认值，可显式覆盖：
    | kind | publish_strategy | apply_policy |
    |------|-----------------|--------------|
    | development | pr-style | require-review |
    | content | overlay | require-review |
    | research | overlay | require-review |
    | office | direct | dry-run |
- **理由**：
  - 正交分解，消除"dry-run 是发布策略"的概念混淆
  - 让"用 pr-style 形态但 dry-run 不真合"这类组合可表达
  - PublishStrategy.apply() 的行为由 apply_policy 驱动（ADR-0030 的 dry-run 语义归到 apply_policy）
- **影响**：
  - 02 架构配置 schema：`publish:` → 拆为 `publish_strategy:` + `apply_policy:`
  - 02 架构 PublishStrategy 表的"默认行为"列拆出
  - ADR-0017 的 kind 默认策略表增列 apply_policy
  - ADR-0030（direct + dry-run apply 语义）改为引用 apply_policy
  - F-6 收尾的"三档默认行为 auto-apply|require-review|dry-run"正式归入 apply_policy
- **风险**：配置字段增多；需在文档里讲清两个字段的区别，避免用户混用

## ADR-0032：框架层组件按"是否需要派别人干活"二分为 skill / subagent
- **日期**：2026-05-17
- **背景**：第十轮验证发现 Claude Code 的全局规则 —— **subagent 拿不到 Agent 工具**，无法再派生 subagent。这击穿了原架构里"Coordinator 作为 subagent 派业务 agent"的核心假设。
- **实测事实**：
  - 主会话 → 项目级 subagent：✅ 可达
  - 主会话 → plugin 内 subagent：✅ 可达
  - subagent → 任何 subagent：❌（frontmatter 写 `tools: [Agent, ...]` 也会被 strip）
  - plugin skill 加载到主会话后保留 Agent 工具：✅
- **选择**：框架层组件按"是否需要派别人干活"二分：
  - **需要派人 → skill / slash command**（运行在主会话上下文，天然有 Agent 工具）
  - **不需要派人 → subagent**（独立上下文，但只能输出文字回主会话）
- **具体分类**：
  | 组件 | 是否派人 | 形态 |
  |------|--------|------|
  | Coordinator | 是 | skill |
  | Scheduler | 是 | skill |
  | Mediator | 是 | skill / slash command |
  | RequirementGatherer | 否 | subagent |
  | Merger | 否 | subagent |
- **理由**：
  - 这是 Claude Code 的硬约束，无法绕开，只能顺应
  - 主会话本来就是天然的编排中枢，让调度逻辑跑在主会话里反而更干净
  - skill 模式下，Coordinator/Scheduler/Mediator 共享上下文，状态传递不需要走文件中转
- **影响**：
  - ADR-0025 的"5 个框架层 agent" → 改为"3 skill + 2 subagent"
  - ADR-0022 的 `Coordinator.handle()` subagent 接口 → 改为 skill 触发约定
  - 02-架构设计的顶层视图、接口规范都要重写
- **风险**：
  - skill 多个共存于主会话时的"角色切换"边界要清晰（不要让一个 skill 把另一个 skill 的指令吃掉）
  - 主会话上下文会被 skill prompt 占用，长任务下要注意 token 预算
- **⚠️ 注意**：理由中"状态传递不需要走文件中转"已被 **ADR-0036** 推翻——上下文预算硬原则要求持久状态必须落 `.agent-team/` 文件，主会话只持摘要。以 ADR-0036 为准

## ADR-0030：apply_policy=dry-run 下 apply() 仅"标记完成"
- **日期**：2026-05-16

## ADR-0029：IsolationStrategy.none 增加产物快照 + 回滚

## ADR-0020：失败处理 = Scheduler 自动重试 N 次后交 Mediator 上报
- **日期**：2026-05-16
- **⚠️ 修订 by ADR-0025 / 0035**：重试 loop 的执行者从 Coordinator 改为 **Scheduler**（ADR-0025 把调度职责从 Coordinator 拆给 Scheduler；ADR-0035 正式定归属并消解三处冲突）。Coordinator 只做派生决策、不碰失败处理；Scheduler 跑 retry loop，超阈值后交 Mediator；Mediator 负责与用户交互。原标题"Coordinator 自动重试"为 stale 表述。
- **背景**：M-5 失败/卡死/超时策略
- **选择**：
  - 默认重试次数 N = 3
  - 重试间隔指数退避（基础 5s, 5s → 10s → 20s）
  - 超时阈值默认 10 分钟无产物更新（agent 可在 md 里覆盖）
  - **Scheduler** 跑重试 loop；超过 N 次仍失败 → 交 **Mediator**，由 Mediator 上报用户，提供选项：重派 / 换 agent / 跳过 / 用户接手
- **理由**：在"立即打扰用户"和"无声失败"之间取平衡；指数退避避免风暴；职责归属与 ADR-0025 的拆分一致
- **配置层**：项目 README team-config 可调整 `retry.max_attempts` `retry.timeout_minutes`

  - **advisory**（默认）：仅在 system prompt 里提示 agent，依赖自觉
  - **strict**：plugin 注册 PreToolUse hook，按 `agent_type` 读对应 files_scope.write，越界写入返回 deny（ADR-0043）
- **理由**：让用户在"低复杂度"和"硬约束"间根据项目重要性选择

## ADR-0018：T3 调度模型 = DAG 默认 + 用户可强制全并发
- **日期**：2026-05-16
- **⚠️ 修订 by ADR-0042**：git-worktree 的"同步上游产物到下游 worktree"表述与 Claude Code 机制不符（worktree 自动建/销毁，塞不进去）。实测模型为"上游分支 merge 进 base → 下游 subagent 的 worktree 在 spawn 时从更新后的 base 派生"。下方"git rebase 到下游 worktree"为 stale 表述，以 ADR-0042 为准。
- **背景**：S-2 worktree 隔离与子任务依赖互斥
- **选择**：
  - 默认 **DAG 调度**：Scheduler 解析 `tasks/<id>.md` 里的 `depends_on` 字段，无依赖才并发
  - 上游完成后，Scheduler 同步上游产物到下游（git-worktree：上游合 base、下游派生，ADR-0042；directory-fork：file copy），再启动下游
  - 用户可在 README team-config 写 `scheduling: parallel-force` 强制全并发（自担合并冲突风险）
- **理由**：DAG 是默认更安全；无依赖的子任务集合下 DAG 自动退化为全并发，无性能损失
- **影响**：必须实现"上游产物同步"机制，不同 IsolationStrategy 实现方式不同

## ADR-0017：跨场景通用 = 抽象出 IsolationStrategy + PublishStrategy
- **日期**：2026-05-16
- **背景**：S-1 自审发现"通用"愿景与 git/PR 偏向冲突，用户选择"首版就跨场景补齐"
- **选择**：
  - **首版覆盖 4 个场景**：软件开发、内容创作、数据/研究调研、办公事务
  - 把"如何隔离 agent 工作"和"如何把成果交给用户"抽象成两个可替换策略：
    - **IsolationStrategy**: `git-worktree` | `directory-fork` | `none`
    - **PublishStrategy**: `pr-style` | `overlay` | `direct`
  - 项目 README team-config 增加 `kind` 字段（apply_policy 拆分见 ADR-0033）：
    - `development` → 默认 git-worktree + pr-style + require-review
    - `content` → 默认 directory-fork + overlay + require-review
    - `research` → 默认 directory-fork + overlay + require-review
    - `office` → 默认 none + direct + dry-run
  - 用户可显式覆盖默认策略
- **理由**：在最早期就用真实场景验证"通用"是否成立，避免做出"号称通用实际只支持开发"
- **风险**：工作量明显增加；每个场景都要走查至少一遍验证抽象立得住

## ADR-0016：agent md 采用 Claude Code subagent 原生格式 + 扩展字段
- **日期**：2026-05-16
- **⚠️ 修订**：被 ADR-0017、0021 部分覆盖（worktree 字段移除、handoff_to 改为 hint）
- **背景**：业务层 agent 的 md 文件 schema
- **选择**：复用 Claude Code subagent 原生 frontmatter (`name / description / tools / model`)，在其上扩展 team 协作所需的字段
- **新增扩展字段（草案，已根据后续 ADR 修订）**：
  - `role`: framework | business
  - `capabilities`: 自然语言描述（被规则匹配器使用）
  - `files_scope.read / files_scope.write`: 可读写的文件范围（权限边界）
  - `handoff_to_hint`: ~~默认完成后交接给谁~~ → 改为下游 hint，真实下游由 Scheduler 决定（ADR-0021）
  - `escalate_to`: 卡住时上报给谁（默认 `mediator`）
  - `triggers`: 规则匹配关键词
  - `quality_gates`: "完成"的判定标准
  - ~~`worktree`: 是否在独立 worktree 工作~~ → **移除**，改为项目 kind 决定 IsolationStrategy（ADR-0017）
- **理由**：与 Claude Code 生态最大化兼容；用户已经熟悉的格式不重新发明
- **风险**：扩展字段如果多到一定程度会失去"原生兼容"的优势 —— 需要克制

## ADR-0015：T4 产出形态 = PR-style 差异 + 项目级默认 + 临时覆盖
- **日期**：2026-05-16
- **⚠️ 修订**：被 ADR-0017 泛化 —— PR-style 现为 PublishStrategy.pr-style 的实现之一，仅适用 development kind；其他 kind 用 overlay/direct
- **背景**：team 完成任务后如何把成果交给用户
- **选择**：默认生成"PR-style 差异 + 变更说明"等用户审核后入主分支；项目 README 可声明默认行为（auto-merge / require-review / dry-run）；用户对话中可临时覆盖
- **理由**：
  - 与 ADR-0014（Worktree 隔离）天然契合 —— worktree 合并产生的差异就是 PR
  - 安全优先：用户保留最终决定权
  - 不破坏既有 git 工作流
  - 三档配置满足从"严谨"到"放手"的不同诉求
- **风险**：每次都审会拖慢节奏 → 必须支持"项目级默认 + 局部例外"的灵活组合

## ADR-0014：T3 并发策略 = Git Worktree 隔离 + 事后合并
- **日期**：2026-05-16
- **⚠️ 修订**：被 ADR-0017 泛化 —— git-worktree 现为 IsolationStrategy.git-worktree 的实现之一，仅适用 development kind；被 ADR-0018 补强 —— 实际调度为 DAG（按依赖），不是简单全并发
- **背景**：多 agent 同时改代码，如何避免冲突
- **备选**：
  - A. 严格串行（同时只一个 agent 写）
  - B. 文件级声明锁
  - C. Git Worktree 隔离 + 事后合并
- **选择**：C
- **理由**：
  - Claude Code 的 Agent 工具原生支持 `isolation: "worktree"` 参数，开箱即用
  - 强隔离意味着 agent 之间真正可以并行，不会互相破坏
  - 与 ADR-0015（PR-style 产出）形成自然管线
- **前提**：项目必须是 git 仓库（非 git 项目需要降级到串行或文件锁）
- **风险**：
  - 每个 agent 看到的是某一时刻快照，与最终合并状态可能有偏差
  - 需要一个"合并 agent" 处理 worktree 间的冲突
  - 增加磁盘 I/O 与目录数量

- **理由**：与 ADR-0009（agents/ 目录）风格统一；天然按文件分隔写权限；可读、可追溯

## ADR-0009：项目业务 agent 走 `.claude/agents/` 目录，每个 agent 一个 md
- **日期**：2026-05-16
- **⚠️ 修订**：目录从"项目根 `agents/`"明确为"项目级 `.claude/agents/`"（第十轮方案 C 验证后定下）。这就是 Claude Code 原生项目级 subagent 位置，主会话自动加载，无需 plugin 干预
- **背景**：项目级配置文件的组织形式
- **备选**：
  - A. 单一 `agents.md`
  - B. `project.md + roles.md + workflow.md` 三件套
  - C. `<dir>/` 目录，每个 agent 一个 md
- **选择**：C，目录路径定为 `.claude/agents/`
- **理由**：
  - 每个 agent 一个文件，符合"动态扩展"特性 —— 加新 agent 只需新增文件，不动其他配置
  - **直接复用 Claude Code 原生项目级 subagent 机制**：Claude Code 启动时自动加载 `.claude/agents/` 下的 md 注册为 subagent，主会话可用 Agent 工具按 name 调用
  - 不需要 plugin 内自我注入 / 用通用 worker 模拟（方案 A/B），架构最干净

- **副作用**：项目级元信息（项目背景、整体工作流）需要另放一处，比如根目录 `README.md` 或 `.claude/agents/README.md`
- **关联**：业务 agent 由主会话（Coordinator skill 上下文）动态调用，ADR-0032 解释为什么调用方必须是主会话

## ADR-0008：agent 间通信采用"共享文件 + 直接调用"组合

  - B. 消息总线 / 任务队列
  - C. 父-子直接调用

## ADR-0006：多入口启动（自然语言 / 项目内 / 斜杠命令）
- **日期**：2026-05-16
- **⚠️ 修订**：被 ADR-0022 补强 —— 三入口统一收敛到 `Coordinator.handle()` 接口；被 ADR-0023 修订 —— "项目内自动激活" 的前提是检测到 `agents/` 目录（无 agents/ 不激活）
- **⚠️ 修订 by ADR-0044**：自然语言入口降级为 best-effort（R-8 demo 实测：skill 自动触发不可靠，主会话可能绕过框架自己干）。**可靠入口 = `/team` 显式指令**；自然语言/自动激活入口仅 best-effort，简单任务主会话直接干也可接受

## ADR-0001：定位为通用 agent team
