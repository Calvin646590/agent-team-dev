## P1 — 必修（3 项）

### P1-1 `SCHED-1`：skip 后下游永远不会 ready → 隐式死锁
**文件**：`skills/scheduler/SKILL.md` 第 60 行、第 195 行  
**问题**：`ready` 条件为 `depends_on 全 done 且自身 pending`；当用户在 Mediator 选择 `skip` 跳过某任务时，该任务状态变为 `skipped`，但 `skipped ≠ done`，所有依赖它的下游任务永远不能进入 `ready` 集合。  
DAG 循环最终触发死锁检测再次升级给用户——但此时没有 `in_progress` 任务，会导致循环死锁报告，而非正常结束。  
第 195 行的"skipped 且下游接受"表述也没有对应的实现路径。  
**修复**：`ready` 条件改为 `depends_on 全 (done 或 skipped)`；Scheduler 完成条件更新为"所有子任务 done 或 skipped"；`blocked` 状态可在 Mediator 将 upstream=skipped 且有 strict 依赖的下游手动标注时使用（不需要自动设置）。

### P1-2 `COORD-1`：`/agent-team init` 命令不存在
**文件**：`skills/coordinator/SKILL.md` 第 41 行  
**问题**：`NoAgentsDir` 错误时提示用户运行 `/agent-team init`，但此命令在 `commands/agent-team.md` 中从未实现——用户执行会得到"无效命令"错误，且没有替代指引。  
**修复**：将提示改为具体的手动操作说明（创建 `.claude/agents/` 目录 + 参考 README 示例）。

### P1-3 `HOOK-1`：`files-scope-guard.py` 中 `os.path.abspath(target)` 未以 `project_root` 为基
**文件**：`hooks/files-scope-guard.py` 第 124 行  
**问题**：当 `target` 为相对路径时（极端情况），`os.path.abspath(target)` 以 Python 进程的 `os.getcwd()` 为基，而非 `project_root`（来自 payload 的 `cwd`）。若 hook 进程启动目录≠项目根，会导致路径解析错误：正常写入被误判为越界拒绝，或越界写入被放行。  
**修复**：改为 `os.path.abspath(os.path.join(project_root, target))`；`os.path.join` 在 `target` 为绝对路径时自动忽略 base，在为相对路径时正确拼接——两种情况均正确。

## P2 — 应修（6 项）

### P2-1 `SCHED-2`：`none_concurrency` 缺省时串行默认未强制
**文件**：`skills/scheduler/SKILL.md` 第 91 行  
**问题**：条件为 `team_config.none_concurrency == "serial"`；若用户 README 的 `team-config` 未配置 `none_concurrency` 字段（office 场景最常见的写法），条件不满足，serial 强制**不生效**——与文档声明的"默认串行"不符。  
**修复**：改为 `none_concurrency 不等于 "file-parallel"`（仅显式 `file-parallel` 时才解除串行约束，缺省视同 serial）。

### P2-2 `SCHED-3`：escalation 中 `files_scope` 数据源标注错误
**文件**：`skills/scheduler/SKILL.md` 第 189 行  
**问题**：注释写"从 task md 读取的 files_scope 字段"，但 task md 文件（`.agent-team/tasks/*.md`）**从不包含** `files_scope`；该字段在 `.claude/agents/<owner>.md` frontmatter 里。注释误导 LLM 去 task md 找一个不存在的字段，导致 `swap_agent` 能力匹配可能用空值判断。  
**修复**：改为"从 `.claude/agents/<task.owner>.md` 读取的 `files_scope.write` 字段"。

### P2-3 `SCHED-4`：Wave 自检是事后断言，逻辑上不可行
**文件**：`skills/scheduler/SKILL.md` 第 143–147 行  
**问题**："检查本轮实际发出的 Agent 工具调用数是否 == len(ready)"——这是对**已发出消息**的自检，但 LLM 无法在消息发出后回溯统计自己的工具调用数。当前写法只能作为意向声明，无法真正检测遗漏。  
**修复**：改为**发出前**的自检清单（预检 pattern）：在构造本轮消息时确认包含 N 个 Agent 调用；发出后记录 log。移除事后检查的"若不等"分支（无法触发）。

### P2-4 `MERG-1`：`task_rollback` 字段描述仅适用 office，非 office 场景语义错误
**文件**：`agents/merger.md` 第 85 行  
**问题**：`task_rollback: <.agent-team/snapshots/<task-id>/ 可一键回...>` 只适用于 office（有写前快照）；development 场景快照目录为空，content/research 场景根本不存在快照。当用户看到 candidate 的 `task_rollback` 字段时，非 office 场景下路径无意义甚至路径不存在。  
**修复**：改为场景分支描述，或标注"仅 office isolation 有效"。

### P2-5 `MERG-2`：Merger step 3 中 `infra` kind 是死代码
**文件**：`agents/merger.md` 第 58 行  
**问题**：`infra: terraform validate / plan --dry-run` — `infra` 不是 Coordinator/Scheduler 任何地方认可的 kind（合法值：development / content / research / office）。该分支永远不会被执行，但会让读者误以为 `infra` 是支持的场景。  
**修复**：删除该行；如未来要支持 `infra`，通过 ADR 正式加入。

### P2-6 `HOOK-2`：~~`hooks/hooks.json` 未在 `plugin.json` 中显式引用~~ — **已核实：无问题**
**核实结论**（2026-05-27，查阅官方文档 code.claude.com/docs/en/plugins）：`hooks/hooks.json` 放在 **plugin root** 是 Claude Code 插件的**标准自动发现路径**，无需在 `plugin.json` 中显式引用。官方文档结构表明确列出："`hooks/` | Plugin root | Event handlers in `hooks.json`"，与 `skills/`、`agents/`、`commands/` 同级，均为自动发现。agent-team 的布局完全正确，**不需要修改**。

## P3 — 建议修（5 项）

### P3-1 `COORD-2`：复杂度闸门 Observer/Commander 区别描述不清
**文件**：`skills/coordinator/SKILL.md` 第 35–37 行  
**问题**：Observer 写"直接建议…问是否仍要 team"，Commander 写"询问用户"——两者行为实质相同（都在问），区别点没有体现。容易让 LLM 对 Observer 模式产生"建议后直接推进"的错误理解。  
**修复**：统一表述为"两种模式均须等待用户明确回应"，消除歧义。

### P3-2 `MED-1`：Mediator Input A schema 缺少 `files_scope` 字段
**文件**：`skills/mediator/SKILL.md` 第 16–24 行  
**问题**：Scheduler step 5 escalation 实际传入 `files_scope`，但 Mediator 的 Input A 约定 schema 没有声明该字段——swap_agent 时要读它，但输入契约里看不到它。  
**修复**：在 Input A schema 补 `files_scope` 字段。

### P3-3 `MED-2`："回传结构化结果给 Scheduler" 机制不显式
**文件**：`skills/mediator/SKILL.md` 第 47 行  
**问题**：Mediator 和 Scheduler 均为 skill（主会话轮流持有），Mediator 完成后 Scheduler 如何恢复并不显式——新手阅读时会疑惑"回传"通过什么机制实现。  
**修复**：补一句说明：Mediator 以结构化输出结束本 skill 调用，主会话重新进入 Scheduler skill 阶段读取结果后继续。

### P3-4 `CMD-1`：`start` 自检说明只涵盖 development 场景
**文件**：`commands/agent-team.md` 自检注释  
**问题**："development 应能看到 worktree 痕迹"——content/research 场景验证信号是 `.agent-team/forks/`，office 场景是 `.agent-team/snapshots/`，均未提及。  
**修复**：补充各场景的验证信号。

### P3-5 `HOOK-3`：`files_scope_enforcement` 检测未限定在 `team-config` 块
**文件**：`hooks/files-scope-guard.py` 第 109 行  
**问题**：`re.search(r"files_scope_enforcement:\s*strict", readme_text)` 扫全文——若 README 注释区或用法示例包含该字符串，会误触发 strict 模式。  
**修复**（低优先级，实际发生概率极低）：先提取 `team-config` 代码块再扫描；或加负向断言 `(?:^|\n)\s*files_scope_enforcement:\s*strict`。
