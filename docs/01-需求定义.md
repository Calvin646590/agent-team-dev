# 需求定义

> 状态：v0.3（2026-05-26 末次更新） · **首版验收标准全部通过**
> 主要变化（v0.2→v0.3）：4 场景端到端全通 + 模式切换验证 + v0.1.10 技术债清零

## 用户故事

### US-1：项目内自然语言启动
> 作为开发者，当我在一个已经配置好 `.claude/agents/` 的项目里说"加个邮件提醒功能"时，agent team 应该自动接管，先和我对齐需求，再分工执行，最后给我一份可审核的变更。

### US-2：模糊需求渐进澄清
> 作为用户，我不需要把任务想得多清楚就能启动。team 里的"需求收集 agent"会通过对话帮我把需求补齐，再开干。

### US-3：动态派生 agent
> 作为用户，我不用预先安排"这个任务派几个 agent"。框架先按规则派，处理不了的模糊场景由协调者用 LLM 判断。

### US-4：参与度可控
> 作为用户，我可以在指挥官 / 旁观者两种模式间切换，让 team 匹配我当下的精力与信任度。（同事/Peer 模式留作 v2，见 ADR-0024）

### US-5：安全收尾
> 作为用户，我希望默认情况下 team 不会直接动我的主分支。它生成一份 PR-style 的差异，我审完决定要不要合。

### US-6：项目级行为可配
> 作为项目维护者，我希望在项目 README 里就能配置默认行为（auto-merge / require-review / dry-run），不需要每次手动指令。

### US-7：跨场景通用（首版即覆盖 4 类）
> 作为用户，同一个 plugin 能服务我的代码项目、内容创作项目、研究调研项目、办公事务项目。项目目录里通过 README 声明 `kind`、定义 `.claude/agents/`，team 自动选择合适的隔离与发布策略。

## 功能需求（必备）

### F-1 入口
- 自然语言触发（Skill 描述命中）—— 仅在项目存在 `.claude/agents/` 目录时激活
- Slash command：`/team start <task>` `/team status` `/team mode <commander|observer>` `/team accept` `/team reject`
- 项目内自动激活：plugin **加载时不能读 cwd**（见 06 调研），改为 skill/command **运行时**检测 `.claude/agents/`（无 `.claude/agents/` → 不激活，提示用户运行 `/team init`）
- 三入口统一触发 Coordinator skill 加载到主会话，按 `{text, source, context}` 协议处理（ADR-0022 / 0032）

### F-2 框架层组件（3 skill + 2 subagent，详 ADR-0032）
- **Skill**（运行在主会话上下文，有 Agent 工具，能派业务 agent）：
  - Coordinator：派生决策（规则 + LLM 兜底 + 缓存）
  - Scheduler：DAG 调度、维护 tasks/index、触发 Merger
  - Mediator：处理 escalate、用户介入、模式切换
- **Subagent**（独立上下文，只输出文字回主会话，不派人）：
  - RequirementGatherer：渐进对话澄清需求
  - Merger：工作区合并、冲突处理

### F-2b 复杂度闸门（ADR-0040）
- Coordinator 派人前先判断任务是否真需要多 agent 协作
- 单 agent 能一次干完的简单任务 → Observer 模式可建议直接单 agent，Commander 模式询问
- 目的：避免"杀鸡用牛刀"，team 的价值在多角色并行/协作

### F-3 派生引擎
- 规则匹配（查 README team-config.derivation_rules）
- LLM 降级（Coordinator 自身判断）
- 决策缓存（.agent-team/decisions.md，双 hash 失效 ADR-0034）

### F-3b 策略层
- IsolationStrategy: git-worktree | directory-fork | none
- publish_strategy: pr-style | overlay | direct（"怎么交付"）
- apply_policy: auto-apply | require-review | dry-run（"要不要 / 怎么应用"，ADR-0033 从 publish 拆出）
- 由 README team-config.kind 决定默认，可显式覆盖

### F-3c 调度模型
- 默认 DAG（按 depends_on）
- 项目可设 `scheduling: parallel-force` 强制全并发

### F-4 任务管理
- `.agent-team/tasks/<id>.md` 子任务文件（owner agent 自行维护进度/产物）
- `.agent-team/tasks/index.md` 索引（**Scheduler** 维护，ADR-0025）
- 子任务字段：id、title、owner、depends_on、next_steps、status、attempts
- 子任务状态机：pending → in_progress → blocked / done / failed

### F-5 通信
- 父子直接调用（任务下发、结果收集）
- 共享文件 `.agent-team/workspace/`（中间产物）
- `handoff_to_hint`（agent md 静态字段）+ `next_steps`（子任务 md 动态字段）→ 由 **Scheduler** 决定真实下游（ADR-0021）
- `escalate_to: mediator` 失败上报（ADR-0020）

### F-6 收尾（按 publish_strategy + apply_policy）
- IsolationStrategy 合并工作区 → 由 Merger 调用
- quality_gates 验证（按各 agent md 声明）
- publish_strategy.assemble() → PublishCandidate
- apply_policy 三档：**auto-apply | require-review | dry-run**（按 kind 默认 + 可覆盖，ADR-0033）
- Commander 模式逐节点审核 / Observer 模式仅关键节点询问

### F-7 用户模式（首版只 2 个）
- Commander / Observer 两种可切换（Peer 留作 v2）
- 项目级默认 + 对话临时覆盖

### F-8 失败恢复
- 自动重试 N 次（默认 3）+ 指数退避
- 超时阈值（默认 10 分钟无产物更新）
- 仍失败 → Mediator 上报用户（重派/换 agent/跳过/接手）

## 非功能性需求

### N-1 可观测性
- `.agent-team/log.md` 完整记录所有 agent 的关键动作（派生、调度、上报、合并）
- 失败时能从日志反推根因

### N-2 安全
- `files_scope` 分级约束（advisory 默认 / strict 可选 hooks 强制）
- 默认收尾不动主分支 / 主文件（PublishStrategy 设默认 require-review）
- 没有项目 README 中显式声明，禁止 push / force-push 等高危操作
- office kind 默认 dry-run，避免不可逆动作

### N-3 性能（方向性，不设硬指标）
- 用户输入到 RequirementGatherer 首条回复要"感觉不卡"
- 不刻意追求极致延迟，优先保证流程正确与可观测
- Coordinator 决策缓存减少重复 LLM 调用

### N-4 易用性
- 错误信息要可操作（含下一步选项），不要只丢栈
- README 模板与 `/team init` 命令降低新项目搭建成本
- 失败时清晰告知"在哪一步、为什么、能选什么"

### N-5 兼容性
- 兼容 Claude Code 现有 subagent 格式（agent md 复用原生 frontmatter）
- 不依赖第三方编排框架
- 项目可以是任意语言/技术栈

## 范围之外（首版不做）

- 跨 agent 的"经验复盘 / 持久学习"
- 远程协作（多人同时操作同一个 team）
- 非 Claude Code Desktop 的运行环境（CLI/Web 等）
- 模板市场 / agent 配置共享
