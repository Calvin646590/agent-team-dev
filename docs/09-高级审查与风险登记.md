# 高级审查与风险登记

> 时间：2026-05-22 · 触发：编码门槛（V-1~V-11）全通过后，进入实际编码前的资深视角复盘
> 目的：把"会让项目失败"的真风险登记在册，先解决再编码。工程留痕，事后可复盘。

## 总体判断

工程纪律强（35 ADR + 13 轮讨论 + 先验证后编码），但**"编码门槛清门"这个说法过于乐观**：

> 已验证的是**积木**（能否调 subagent / 并发 / hook / 接力），不是**建筑**（一个真实多 agent 任务从 T0 跑到 T4）。最大风险不在已验证的原语，而在**尚未验证的集成**，以及 **C' 架构（主会话手动跑多 agent 状态机）在真实规模下是否稳定**。

心理预期应从"技术已证、剩体力活"调整为"原语已证、真正的架构验证才刚开始"。

## 优势（保持）

1. 诚实的失败处理（V-3 击穿后直接转 C'，不粉饰）
2. 可回溯性极强（ADR + 讨论记录）
3. 策略抽象（Isolation/Publish 两正交维度统一 4 场景）概念立得住
4. 先验证后编码的工程习惯

### R-8：框架被主会话绕过（2026-05-23 demo 实测发现，🔴🔴 最高）

**现象**：development demo 跑了两次，**两次都没进框架**。实证：
- agent-team plugin 已安装 + 启用 + 3 skill 都在、coordinator 描述含触发词"用 team 加个 X 功能"
- 但主会话收到"用 team 给 string-utils 加 slugify…"后，**直接读项目 + 并行派 3 个业务 agent**，没加载 coordinator skill
- demo 项目无 `.agent-team/`、无 worktree、改动直接落主工作区（无 Merger / pr-style）—— Coordinator/Scheduler 根本没跑
- 且并行派发**违反了 depends_on**（tester 与 developer 并行 → tester 启动时无 slugify，撞竞态，靠侥幸蒙对）

**根因**：**skill 自动触发是建议性的，主会话有完全自主权**。对它一眼能直接做的简单任务，它判断"不需要编排 skill"就自己干了；"用 team"被理解成"用多 agent"而非"加载 coordinator 协议"。

**为何比 R-2 更深**：R-2 是"LLM 把状态机跑歪"；R-8 是"**LLM 根本不进状态机**"。整个架构假设"主会话会加载并遵守框架 skill"——经验上对自然语言入口 + 简单任务**不成立**。

**与 ADR-0040 的张力**：复杂度闸门本就说"简单任务别用 team"。所以主会话绕过重型编排某种程度上合理——但它绕过时连**最低限度的依赖意识**都丢了（撞竞态）。

**修复方向（待定）**：
1. **explicit `/team start <task>` slash command** 作为可靠入口——显式调用 + 命令体强指令"现在执行 agent-team 编排，按 Coordinator 协议逐步走，不要自己直接做"。比被动 skill 描述匹配强得多
2. 自然语言入口降级为 best-effort（承认它不可靠；简单任务主会话直接干也 OK）——可能要修订 ADR-0006/0022 的自然入口预期
3. 重测要用**真正需要协作的任务**（真实依赖/多文件/有冲突可能），否则复杂度闸门会正确地判定"不必上 team"
4. 即便不走全套，主会话也应有最低依赖意识（先 developer 再 tester）

**状态**：✅ **已消化**（2026-05-24，ADR-0044/0045）：`/agent-team start` 强制入口 + plugin 内 `commands/agent-team.md` 硬约束"不得自己直接做"。4 场景实测（2026-05-26）全程正确进入框架，无一绕过。

### R-1：主会话上下文预算（🔴 系统性）

**问题**：Coordinator/Scheduler/Mediator 都跑在同一主会话上下文；三 skill 接力 → 三份大 prompt 累加；Scheduler 的 DAG/retry 循环全在主会话；**所有 subagent 返回都回流主会话**。一个 10 子任务 + 重试的真实任务，主会话上下文很可能爆。V-10 只测了 3 个返回一行字的 sleep-worker，没暴露这点。

**矛盾点**：coordinator.md L91 现在写"状态传递不必走文件中转"——方向是反的。

**解决方向**：
- 立"**状态尽量落 `.agent-team/` 文件、主会话只持指针/摘要**"为硬原则（新 ADR）
- subagent 返回也落盘（写到自己的 task md / workspace），主会话只读摘要，不把全文吞进上下文
- 改 coordinator.md L91 的反向表述
- demo 时实测主会话上下文增长曲线

---

### R-2：LLM 驱动的状态机 = 没有真正的状态机（🔴 结构性）

**问题**：Scheduler 的 DAG 解析、attempts 计数、status 流转、retry 退避，全是 prompt 指令靠 LLM 忠实执行，没有一行代码强制不变量。任务越大，跑偏（漏 ready 节点 / 重复派 / status 算错 / retry 多跑）概率越高。对"框架"是结构性脆弱。

**解决方向（需决策，三选一）**：
- **A. 首版接受 prompt 驱动** + 加"状态自检"步骤（每轮循环重读 index.md 校验不变量）+ demo 重点观察可靠性 + 登记为技术债
- **B. 把确定性内核抽成代码**（脚本 / 轻量 MCP server：DAG 解析、status 流转、retry 计数），主会话调它而非心算
- **C. 混合**：首版 A，demo 暴露不稳后再做 B
- 倾向 **C**：先用 A 跑 demo，用真实数据决定要不要上 B，避免过早工程化

### R-3：集成风险被验证假象掩盖（🔴 验证缺口）

**问题**：V-1~V-11 验的是单点原语。真正难的组合从没测：
- **并发 + worktree 同时**：V-10 测并发（无隔离 worker），V-4 测 worktree（单个）。但 T3 是"多 subagent 各自 worktree 并发 + 完成后 rebase 上游产物到下游 worktree"——组合 + 跨 worktree 同步**未验**，且是开发场景核心
- **端到端 T0→T4**：一个真实任务跑完整链路**从没跑过**
- **strict hook 的 per-agent scope**：V-6 测的是全局 tripwire；真实 strict 要"按当前是哪个 agent 应用其 files_scope"——**hook 能否拿到当前 subagent 身份，未验**，可能根本拿不到

**解决方向**：
- 新增 **V-12**：多 subagent 各自 worktree 并发 + 跨 worktree 同步（同步机制待实测——后由 V-12 确立为"上游合 base→下游派生"，ADR-0042）
- 新增 **V-13**：PreToolUse hook 能否识别"当前哪个 subagent 在调用工具"（决定 strict 模式 per-agent scope 是否可行——后由 V-13 确立可行，ADR-0043）
- **端到端最小 demo** 作为阶段三第一步，而非"把草稿变 plugin"

### R-4：4 场景同时上（🟡 范围）

**问题**：首版就要 4 场景 + 3 Isolation + 3 Publish + Merger 3 套冲突策略，每套都未经集成验证。第八/十一轮的"走查"是纸面推演，不是跑通。

**解决方向（需决策）**：
- 编码阶段**先把一个场景端到端跑透**（建议 development：worktree/pr 最成熟），再复制模式到其他三个
- 不四个并行铺开
- 登记为执行顺序约定（新 ADR 或写入 README 路线）

---

### R-5：Merger directory-fork"末端 fork"假设（🟡 设计）

**问题**：merger.md 写"主线产物取流水线末端 fork 版本"，只在线性 DAG 成立。DAG 分叉（两 writer 各写不同章节后合并）时"末端 fork"无定义。

**解决方向**：
- 改为**显式声明产物归属**：子任务 md 增 `outputs:` 字段，agent 完成时声明自己产出哪些文件；Merger 按声明归集，不靠"末端"推断
- 同名文件多 agent 写 → 仍标 `conflict: manual`
- 改 merger.md + 02 架构对应段

---

### R-6：没有"何时值得动用 team"的阈值（🟡 设计）

**问题**：每个子任务 = 一次 subagent spawn = 一次完整 LLM 上下文。简单任务用整套编排，开销可能远超单 agent 直接做。office"整理个表格"可能不值得 5 组件编排。

**解决方向**：
- Coordinator 第 1 步增加"**复杂度闸门**"：若任务可由单 agent 一次完成（无需多角色协作）→ 直接提示"这个任务用单 agent 更划算，要继续用 team 吗？"或直接单 agent 干
- 写入需求（新 F 项或 N 项）+ coordinator.md

---

### R-7：跨子任务事务性（🟡 设计）

**问题**：单 isolation 内有 snapshot 回滚，但整个任务跑到一半失败（office direct 已 apply 3 个、第 4 个炸）如何回到一致状态？跨子任务一致性/回滚故事薄。

**解决方向**：
- T4 apply **批量化 + 尽量后置**：所有子任务产物先 assemble 成 candidate，用户一次性 review/apply，避免"边跑边 apply"造成半应用
- office direct 默认 dry-run 已部分缓解（产物在 outputs/，apply 才动外部）
- 定义**任务级回滚**：记录本次任务所有 snapshot，失败时可一键回滚到任务开始前
- 写入 02 架构失败处理 + ADR

## 元观察：过度设计 vs 不足

- **可能过度**：零产品代码前铺了 35 ADR / 5 组件 / 4 场景 / 3+3 策略，有分析瘫痪苗头
- **可能不足**：上下文预算、LLM 调度可靠性、集成测试——真正会让项目失败的点着墨最少
- 典型"把容易的事做透、把难的事推后"

---

## 推荐的总路线修正

不要直接进"把 5 草稿变可分发 plugin"。改为：

1. **先做设计层修复**（R-1/R-4/R-5/R-6/R-7，纯文档/ADR，当下可做）
2. **再补集成验证**（R-3 → V-12/V-13；R-2 的可靠性观察）
3. **端到端最小 demo**（development 单场景，T0→T4 跑通 + 观察 R-1/R-2/R-3）
4. demo 过了、风险消化了，**才**进入"4 场景铺开 + 正式 plugin 化"

> 解决计划的逐项排期见对话 / 后续讨论记录。
